Step 3 看 Service 怎么把策略跑起来 8 分钟

看哪里

src/services/fwflashservice.cpp

  • 第 74–113 行 startFlash()——查策略、设状态、调前置序列
  • 第 131–223 行 runPreflightAndFlash()——本步重点

重点看 3 件事

  1. worker 线程怎么起：QThread::create([self, strategy, firmware, cancelFlag, total]() { ... })
    • firmware 用 QByteArray 隐式共享，不会真拷贝
    • cancelFlag 是 shared_ptr，引用计数 +1，worker 比 service 死得晚也安全
    • self 是 QPointer<FwFlashService>，防御 service 提前析构
  2. worker 里怎么 emit 信号回 UI：emit self->progressUpdated(...)——跨线程时 Qt 自动转 QueuedConnection
  3. 结果怎么回主线程改状态：QMetaObject::invokeMethod(self.data(), [...]() { setState(Completed); }, Qt::QueuedConnection)
易混淆点：setState() 只允许在主线程调用——这就是 worker 完成后必须用 invokeMethod 把状态切换"打回主线程"的原因。不要在 worker 里直接调 setState。

自检问题

  • worker 线程结束后是谁负责删除 QThread*？查 connect 那两行。
  • 如果用户在前置序列阶段就点了取消，会发生什么？看第 152–160 行。

Step 6 理解 thunk 路由（难点 #1） 10 分钟

看哪里

aw_sdk_strategy.cpp 第 262–279 行 三个 thunk + 第 34 行 thread_local s_currentInstance + flash() 第 67、72、106、107 行 的赋值与清零。

核心问题：为什么需要 thunk？

  • Awinic DLL 的 AwOIS_ExtFuncInit 接受一个 DllExtFuncList 结构，里面是 3 个 C 风格函数指针
  • C 风格函数指针不能指向成员函数（成员函数有隐式 this）
  • 所以必须用静态函数当跳板（thunk）。但静态函数怎么知道 "我属于哪个 strategy 实例"？
  • 答案：用一个静态指针变量记住当前正在运行的实例——这就是 s_currentInstance

为什么是 thread_local？

  • 普通静态变量是全局共享的——如果将来有两个 worker 同时跑两个 strategy，会相互覆盖
  • thread_local 让每个线程有一份独立副本，多线程也安全
  • 当前架构虽然只有一个 worker，但 thread_local 是廉价的防御

赋值时机

  1. flash() 入口第 67 行：s_currentInstance = this;
  2. 正常退出第 106 行：s_currentInstance = nullptr;
  3. 失败退出 lambda bail() 第 71 行：同样清零
易混淆点：thunk 不是"发起调用"的人，而是"被 DLL 调用"的人。方向是 DLL → thunk → 实例方法，不是反过来。看清这一点，整个回调机制就通了。
陷阱：如果你将来重构，把 s_currentInstance 改成普通成员变量，那就彻底跑不通——因为 C 风格函数指针签名里没有任何地方携带 this。

自检问题

  • 如果 s_currentInstance 在 thunk 调用时是 nullptr，会发生什么？看第 264–266 行。这种情况什么时候可能发生？
  • 把 thunk 三件套画成时序图：DLL 调用 thunk → thunk 取实例 → 实例方法返回 → DLL 拿到结果。

Step 7 实例转发层 5 分钟

看哪里

aw_sdk_strategy.cpp 第 285–307 行 onReadI2c / onWriteI2c / onOutputLog。

每个方法看 2 件事

  1. 取消优先：方法第一句就检查 m_cancelFlag，true 直接返回非零。这会让 DLL 主动中止整段下载。
  2. onWriteI2c 的进度联动：写成功 + m_inDownload 才累计。前面 init/boot 阶段的写不算进度——因为那些字节不是固件。

关于 addrSize 的特殊约定

第 295–296 行注释非常重要——DLL 把"寄存器地址 + 数据"拼成单段交给我们，所以 onWriteI2c 转发到 syncI2cWrite 时 固定填 addrSize=0。STM32 端按"无地址前导，全部当作数据"模式直接发送。

自检问题

  • onReadI2c 为什么不需要"取消时 DLL 自动中止"机制？提示：读操作通常是单次的，不在循环里。
  • onOutputLog 直接把 DLL 字符串转 UTF-8 显示——如果 DLL 输出的是 GBK 怎么办？（潜在缺陷，记下来。）

Step 8 同步透传核心（难点 #2） 15 分钟

看哪里

aw_sdk_strategy.cpp 第 313–322 行 I2cWaitState 结构 + 第 326–422 行 syncI2cWrite / syncI2cRead。

建议读法（重要）

  1. 先只读 syncI2cWrite，不要碰 syncI2cRead。两者结构 95% 相同，搞懂一个另一个就通了。
  2. 把 326–368 行的 43 行代码分成 4 段逐段理解。

4 段拆解

段行号做什么关键设计
① 协议编码 333–334 调 encodeI2cTransferWrite() 把参数打包成命令字 0x30 的载荷 编码逻辑封装在 motor_protocol，本层不关心字节布局
② 等待状态 336–346 建一个 shared_ptr<I2cWaitState> 和回调 lambda shared_ptr 是关键：即使 worker 提前因取消返回，dispatcher 回调仍能安全写 state
③ 跨线程提交 348–353 invokeMethod + QueuedConnection 把 submitCommand 投到 dispatcher 所在线程 worker 不能直接调 dispatcher（线程关系不允许）
④ 阻塞等待 355–367 QWaitCondition::wait(timeoutMs) 在 worker 上挂起，被回调 wakeOne() 唤醒 循环 wait 是为了防虚假唤醒；唤醒后再次检查 cancelFlag 提前退出

这一步最难的概念：QMutex + QWaitCondition

如果你之前没用过条件变量，务必先理解这个模式：

// 等待方（worker）
QMutexLocker locker(&mutex);
while (!ready) {
    cond.wait(&mutex, timeoutMs);   // 释放锁 + 阻塞 + 被唤醒后重新拿锁
}

// 通知方（回调）
QMutexLocker locker(&mutex);
ready = true;
cond.wakeOne();                       // 唤醒一个等待者

关键点：cond.wait() 内部会原子地"释放锁 + 进入阻塞"，醒来时原子地"重新获得锁"。这避免了"判断 ready 和 wait 之间被插入 wakeOne"的丢失唤醒问题。

易混淆点：第 355 行的 QMutexLocker locker(&state->mutex) 和回调里的 QMutexLocker 用的是同一把锁（同一个 state）。两个线程不会同时持有这把锁。
陷阱：如果把第 355 行删掉直接 cond.wait(...)，会立刻 crash——QWaitCondition::wait 要求传入已锁定的 mutex。

读 syncI2cRead 时只关注差异

3 处不同：

  1. 编码用 encodeI2cTransferRead（命令字 0x31）
  2. 回调里多一句 state->data = respData; 把响应数据存下来
  3. 等到响应后多两步：长度校验、memcpy 到 DLL 给的 outBuf
读完应该能回答：为什么 syncI2cWrite 的回调没用 respData，而 syncI2cRead 用了？因为写不需要返回值，只需要"成功 / 失败"。

自检问题

  • 如果 STM32 永远不回响应，最坏情况 worker 会卡多久？看 kI2cTimeoutMs 常量。
  • 如果用户在等响应中途点了取消，哪一行让 worker 提前退出？提示：第 361 / 407 行。
  • 为什么 state 用 shared_ptr 而不是栈对象？答案：dispatcher 回调可能在 worker 已退出后才到达；如果 state 是栈对象就成野指针了。
