从打开 HEX 到烧录完成 — 当前代码的完整执行路径

项目：MotorDev  |  范围：src/tabs/fwflashtab · src/protocol/firmware_parser · src/services/fwflashservice · src/services/flashstrategies/aw_sdk_strategy · src/protocol/motor_protocol

本文按代码当前实际行为逐步描述，从用户在 UI 上点击「浏览...」选择 .hex 文件，到一次完整烧录序列结束。 每一步都标注 file:line，便于跳转源码核对。

零、整体阶段一览

[1] 选择文件         FwFlashTab::onBrowseClicked          (主线程, UI)
[2] 解析固件         FirmwareParser::parseFile            (主线程, 同步)
[3] 显示文件信息     FwFileInfoPanel::setInfo             (主线程, UI)
[4] 点击开始         FwFlashTab::onStartRequested         (主线程, UI)
[5] 前置停止序列     FwFlashService::runPreflightAndFlash (主线程, fire-and-forget)
[6] worker 线程烧录  QThread + AwSdkStrategy::flash       (worker 线程)
[7] DLL 5 步流程     ExtFuncInit → Bootcontrol →          (worker 线程)
                     MoveBinDownload → ResetChip → DeInit
[8] I2C 透传回调     onWriteI2c → syncI2cWrite (cmd 0x30) (worker → dispatcher)
[9] 完成/失败/取消   invokeMethod 回主线程切状态           (worker → 主线程)

一、选择文件（UI 主线程）

用户点击「浏览...」按钮触发 QFileDialog，过滤器只允许 .bin 与 .hex。 选中后立即调用 parseAndShowFile(path) 把后续工作交给协议层。 

void FwFlashTab::onBrowseClicked() {
    const QString path = QFileDialog::getOpenFileName(
        this,
        tr("选择固件文件"),
        QString(),
        tr("Firmware Files (*.bin *.hex);;Binary (*.bin);;Intel HEX (*.hex)"));
    if (path.isEmpty()) return;
    parseAndShowFile(path);
}

来源：src/tabs/fwflashtab.cpp:324

二、解析固件文件（协议层）

FirmwareParser::parseFile(path) 是一个静态工具方法，返回一个值类型 FirmwareInfo。 对 .hex 文件，它做的事不仅是逐行解析，还包括段合并与CRC32 计算， 下游 UI 和烧录策略都直接使用 FirmwareInfo::data，不再二次处理。

字段含义谁会读
valid / errorMessage解析是否成功 + 失败原因UI 三页切换
data合并后的连续二进制（hex 段间用 0xFF 填充）策略层烧录原料
crc32对 data 做 IEEE 802.3 CRC32UI 显示 + 日志
segments / minAddress / maxAddresshex 段元信息UI 显示
fileSizeBytes / effectiveBytes原文件大小 / 有效数据字节数UI 显示

约束：

  • 固件大小上限 1024 KB（FirmwareParser::MaxFileBytes），合并后超过即视为非法。
  • Intel HEX 支持类型 04（高 16 位扩展地址），段间用 0xFF 填充对齐。
  • 解析失败时其他字段无意义，只有 errorMessage 可用。

来源：src/protocol/firmware_parser.h:36-64

三、显示文件信息（UI）

parseAndShowFile 拿到 FirmwareInfo 后，根据 valid 切换三种 UI 状态， 并把 data 拷贝到成员 m_currentFirmwareData（后续启动烧录时直接使用）。 

void FwFlashTab::parseAndShowFile(const QString &path) {
    m_currentFilePath = path;
    m_pathEdit->setText(path);

    const FirmwareInfo info = FirmwareParser::parseFile(path);
    m_infoPanel->setInfo(info);          // FwFileInfoPanel 内部 QStackedWidget 切页

    if (!info.valid) {
        m_currentFileValid = false;
        m_currentFirmwareData.clear();
        m_currentFirmwareTotal = 0;
        m_logPanel->appendError(tr("文件解析失败：%1").arg(info.errorMessage));
    } else {
        m_currentFileValid = true;
        m_currentFirmwareData = info.data;
        m_currentFirmwareTotal = info.data.size();
        m_logPanel->appendInfo(
            tr("已加载固件：%1（%2 字节，CRC32 0x%3）")
                .arg(info.fileName).arg(info.data.size())
                .arg(info.crc32, 8, 16, QLatin1Char('0')));
    }
    updateStartEnabled();                // 同步「开始」按钮可用状态
}

updateStartEnabled() 的启用条件：IC 已选 + 文件有效 + 服务非 busy。 任一不满足，「开始」按钮保持禁用。

来源：src/tabs/fwflashtab.cpp:339-398

四、点击「开始」触发烧录服务

FwFlashControlPanel 的「开始」按钮 emit startRequested， 转到 FwFlashTab::onStartRequested()，再下发到 FwFlashService::startFlash()。 

void FwFlashTab::onStartRequested() {
    const QString icModel = m_icCombo->currentData().toString();
    if (icModel.isEmpty() || !m_currentFileValid) {
        m_logPanel->appendWarn(tr("请先选择 IC 与有效固件文件"));
        return;
    }
    m_service->startFlash(icModel, m_currentFirmwareData, m_currentFirmwareTotal);
}

FwFlashService::startFlash() 做四件事：

  1. 检查 isBusy()、m_registry、IC 注册情况、固件非空，任一失败立刻 finished(false, ...) 返回。
  2. 回收旧 worker 线程（防御性，正常路径下应已退出）。
  3. 创建新的 std::shared_ptr<std::atomic<bool>> m_cancelFlag，初值 false。
  4. 切状态 Preparing，发首条 stageMessage + 进度 0，进入 runPreflightAndFlash()。

来源：src/tabs/fwflashtab.cpp:404-411 · src/services/fwflashservice.cpp:74-113

五、前置停止序列（主线程，fire-and-forget）

烧录前需要保证 STM32 端处于安静状态：停掉示波器采样、停掉波形发生器、停掉循环写入、关掉 PMIC 输出。 这四个动作通过 FwFlashTab 在构造时由 MainWindow 注入的 std::function<void()> 回调触发。 

// runPreflightAndFlash() 摘要
setState(StoppingScope);     invokeIfSet(m_stopScope);
setState(StoppingGenerator); invokeIfSet(m_stopGenerator);
setState(StoppingCyclic);    invokeIfSet(m_stopCyclic);
setState(DisablingPmic);     invokeIfSet(m_disablePmic);

if (m_cancelFlag->load()) {
    // 前置期间用户已点取消 → 直接 Cancelled 收尾
}
fire-and-forget 语义：四个回调中任何一个未注入或失败，Service 只写日志，绝不阻断后续烧录。 这是产品决策——不假设这些远端动作必须成功，且不强等 ACK。

来源：src/services/fwflashservice.cpp:131-160

六、进入 worker 线程执行烧录

前置序列完成且未取消后，Service 切到 Flashing 状态， 用 QThread::create 启动一个一次性 worker 线程，把 strategy->flash(...) 放进去跑。 

m_workerThread = QThread::create([self, strategy, firmware, cancelFlag, total]() {
    QString errorMsg;
    auto progress = [self, total](qint64 sent) {
        // 跨线程 emit，AutoConnection 自动转 QueuedConnection 到主线程
        emit self->progressUpdated(sent, total);
        emit self->stageMessage(QStringLiteral("烧录中 %1 KB / %2 KB (%3%)")...);
    };
    const bool ok = strategy->flash(firmware, progress, *cancelFlag, &errorMsg);
    const bool wasCancelled = cancelFlag->load();

    QMetaObject::invokeMethod(self.data(), [self, ok, wasCancelled, errorMsg, total]() {
        // 在主线程切 Completed / Failed / Cancelled 并 emit finished
    }, Qt::QueuedConnection);
});
m_workerThread->start();

注意两点：

  • QPointer<FwFlashService> self 防御 service 在 worker 完成前被销毁。
  • cancelFlag 是 shared_ptr 拷贝，保证 worker 内对原子标志的访问始终安全。

来源：src/services/fwflashservice.cpp:162-223

七、AwSdkStrategy 五步流程（worker 线程内同步串行）

AwSdkStrategy::flash() 是 AW86006 / AW86100 共用的烧录算法， 通过 QLibrary 动态加载厂家 DLL（默认 AW86100.dll），按固定顺序调用 5 个导出符号。

步骤 0：加载 DLL 与符号 resolve

ensureDllLoaded() 先 m_library.setFileName(dllPath()) + load()， 然后逐一 resolve 5 个符号；任一缺失立即失败并写 errorMessage。 

resolve: AwOIS_ExtFuncInit
         Aw_OIS_ExtFuncDeInit
         AwBootcontrol
         AwMoveBinDownload
         AwResetChip

步骤 1：AwOIS_ExtFuncInit

把 3 个 C 风格回调（thunk）注册给 DLL： 

DllExtFuncList list = {
    &AwSdkStrategy::extReadI2cThunk,    // pFunc_I2C_Read
    &AwSdkStrategy::extWriteI2cThunk,   // pFunc_I2C_Write
    &AwSdkStrategy::extOutputLogThunk,  // pFunc_OutputLog
};
const int ret = m_fnInit(&list);        // 0 视为成功

成功后置 m_initCalled = true（决定收尾路径是否需要 DeInit）。 Thunk 通过静态 thread_local s_currentInstance 路由回当前 strategy 实例方法。

步骤 2：AwBootcontrol(20, 7)

让 IC 进入 boot 模式，参数 (times=20, delay=7) 是厂家给定默认。 成功后置 m_bootEntered = true（决定收尾路径是否需要 ResetChip）。

步骤 3：AwMoveBinDownload(buf, len, 0)

把 QByteArray 的裸指针交给 DLL，DLL 内部循环切包并通过回调下发到 I2C 总线。 第 3 参 kDLNumType_0（16 word per pack）已由用户决议固定为 0。

下载阶段 m_inDownload = true，每次 onWriteI2c 成功后累计字节并节流上报进度。

步骤 4：AwResetChip

复位 IC，使其从 boot 模式退出回到正常运行模式。成功后清 m_bootEntered。

步骤 5：Aw_OIS_ExtFuncDeInit

释放 DLL 内部资源。返回值若非零仅写 warn 日志，不报失败（已经走到这一步意味着核心流程完成）。

取消与异常收尾：每步前 checkCancel()，任一步失败走 emergencyCleanup()： 若已进 boot 强制调 AwResetChip；若已 Init 强制调 DeInit。 目的是让 IC 不被卡在 boot 模式。收尾本身的失败不覆盖原始 errorMessage，仅追加日志。

来源：src/services/flashstrategies/aw_sdk_strategy.cpp:53-256

八、I2C 透传回调（下载阶段循环触发）

下载阶段，DLL 内部每打包一帧数据就回调 I2C_Write 一次。完整调用链： 

DLL  ─►  extWriteI2cThunk(devId, wrSize, wrData)        // C-style, 无 this
       │   通过 thread_local s_currentInstance 路由
       ▼
       onWriteI2c(devId, wrSize, wrData)                 // 实例方法
       │   检查 cancelFlag；按 addrSize=0 调
       ▼
       syncI2cWrite(devId, 0, nullptr, wrSize, wrData, 5000ms)
       │   1) MotorProtocol::encodeI2cTransferWrite()    // cmd 0x30
       │   2) QMetaObject::invokeMethod(dispatcher, ..., QueuedConnection)
       │      dispatcher 在主线程 submitCommand(High, callback)
       │   3) QMutex + QWaitCondition 阻塞等待响应
       │   4) STM32 回帧 → CommandDispatcher 触发 callback
       │      检查是否为 CmdErrorResponse / LocalErrorCode
       │   5) wakeOne() 解除阻塞
       ▼
       返回 0 → notifyProgress(wrSize) 累计字节并节流上报

关键设计：

  • 同步阻塞：DLL 的 I2C 回调必须同步返回结果，但我们的串口通信是异步的， 所以 worker 线程用 QWaitCondition 把异步响应"压"成同步语义。
  • 跨线程提交：CommandDispatcher 生存在主线程，worker 通过 invokeMethod + QueuedConnection 把 submitCommand 转过去执行。
  • AddrSize == 0：寄存器地址段交给 DLL 自己拼进 wrData， STM32 端按 HAL_I2C_Master_Transmit 直接发送整段 data（地址段跳过）。
  • 取消传播：等待循环每次 wait 后再检查 cancelFlag， 标志翻 true 立刻返回 -1，DLL 收到非零会主动中止下载。
  • 超时 5s：单次 I2C 透传最长等 5000ms，超时返回 -1。

进度上报节流：notifyProgress() 不是每包都 emit，而是按 16ms 间隔 或 每 1KB 二者满足其一才上报，避免 UI 线程被频繁刷屏。

读操作 syncI2cRead（cmd 0x31）机制对称，多一步把响应 memcpy 进 DLL 的 pRdBuf，并校验返回长度不小于请求长度。

来源：src/services/flashstrategies/aw_sdk_strategy.cpp:262-422

九、完成 / 失败 / 取消收尾

worker 线程内 flash() 返回后，结果通过 QMetaObject::invokeMethod 切回主线程：

结果状态切换UI 表现
成功（ok == true） Flashing → Completed → Idle 进度条 100%、stage "烧录完成"、Ok 日志、finished(true, "成功")
取消（cancelFlag == true） Flashing → Cancelled → Idle stage "已取消"、Info 日志、finished(false, "已取消")
失败 Flashing → Failed → Idle stage "失败：xxx"、Error 日志、finished(false, errorMsg)

FwFlashTab 的 onServiceFinished 槽再调一次 updateStartEnabled()， 按钮、Combo、Browse / Clear 恢复可用。

来源：src/services/fwflashservice.cpp:191-223 · src/tabs/fwflashtab.cpp:442

附录 A：协议帧（I2C 透传 0x30 / 0x31）

烧录期间所有底层 I2C 通信都封装为以下两个命令码：

命令方向编码函数含义
0x30 CmdI2cTransferWrite 上位机 → STM32 MotorProtocol::encodeI2cTransferWrite 同步 I2C 写：任意 DevId + AddrSize 字节地址 + DataLen 字节数据
0x31 CmdI2cTransferRead 上位机 → STM32 MotorProtocol::encodeI2cTransferRead 同步 I2C 读：任意 DevId + AddrSize 字节地址 + 期望读取长度

本场景下 AddrSize == 0，地址段由 DLL 拼在 wrData 内，STM32 端跳过寄存器地址段直发 data。 该 HAL 调法目前按推测实现，等供应商最终确认（参见 docs/file_index.md OPEN 项）。

附录 B：线程与同步要点

  • FwFlashService 自身：生存于主线程，m_state 仅在主线程修改。
  • 前置停止序列：在主线程同步完成，回调内部行为由各 Service 自行决定（多半是 invokeMethod 转到 SerialManager 线程）。
  • strategy->flash()：在 worker 线程跑，期间所有 emit signal 走 Qt AutoConnection 自动转 QueuedConnection 回主线程。
  • CommandDispatcher：生存于主线程，worker 通过 invokeMethod 跨线程 submit。
  • SerialManager：在独立线程，但本流程不直接调它，全部走 dispatcher 中转。
  • cancelFlag：shared_ptr<atomic<bool>>，主线程置位，worker 与各 I2C 等待循环查询。

附录 C：常见失败点速查

失败位置典型表现处理方式
FirmwareParser 解析 UI 切到错误页 + 红色日志，「开始」按钮保持禁用 用户重新选择文件即可，无副作用
DLL 加载或符号 resolve "加载 DLL 失败"或"DLL 缺少导出符号" 直接失败返回，无 boot 状态需要回滚
AwBootcontrol 失败 IC 未进入 boot 模式，DLL 返回非零 仅做 DeInit；IC 仍在正常模式，无需 ResetChip
AwMoveBinDownload 失败 I2C 中途失败或 DLL 校验失败 emergencyCleanup 强制 ResetChip + DeInit，IC 退出 boot
I2C 透传超时（5s） STM32 长时间未回帧 onWriteI2c 返回 -1，DLL 中止；emergencyCleanup 收尾
用户点取消 cancelFlag 翻 true，I2C 回调返回 -1 让 DLL 中止 emergencyCleanup 收尾，UI 切 Cancelled
本文描述的是当前代码现状，覆盖 AW86006 / AW86100 通过 AwSdkStrategy + dummy DLL 联调的完整链路。 DW9786 / DW9788 仍为 stub，与本流程无关。
