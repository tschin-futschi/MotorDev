I2C 透传功能 — 单步调试指南

工具链：Qt MinGW 13.1.0 64-bit（项目自带） |  调试器：GDB（MinGW 内置） |  推荐 IDE：VSCode + C/C++ 扩展（cppdbg）

调试这部分代码的两大难点：
  1. 跨线程——一次 I2C 透传至少切换 3 个线程（worker → 主线程 → SerialManager 线程）；单步键按下去，线程焦点会跳。
  2. DLL 边界——Awinic DLL 没有源码，AwMoveBinDownload 内部 step into 会进汇编。需要"在边界打断点而不是 step 进去"。
本指南给一套可执行的方案：从构建、launch.json、断点策略、跨线程接力、到 wait/wake 这种"看起来卡住"的代码该怎么处理。

路径总览

  1. Step 1 — 准备 Debug 构建（带调试符号）
  2. Step 2 — 配置 VSCode launch.json（最关键）
  3. Step 3 — 设置 Qt printers，让 QString 等可读
  4. Step 4 — 选硬件还是模拟器（决定调试源）
  5. Step 5 — 战略断点 9 个固定位置
  6. Step 6 — VSCode 调试面板使用要点（线程切换、Watch、Conditional BP）
  7. Step 7 — 跨线程"接力调试"模板
  8. Step 8 — DLL 边界与 QWaitCondition::wait 的处理
  9. Step 9 — 不打断的诊断手段（logpoints、qDebug、Qt 日志类别）
  10. Step 10 — 常见症状速查表

Step 1 准备 Debug 构建（带调试符号） 5 分钟

问题

当前 build_make_qt/CMakeCache.txt 里 CMAKE_BUILD_TYPE 是空——意味着用了默认配置，可能没有 -g，单步进去会"乱跳"或者完全看不到源码。

方案 A（推荐）：单独建 Debug 构建目录

不污染原 Release 目录，可以并存切换。

# 在项目根目录打开 PowerShell
$env:PATH = "D:\Qt\Tools\mingw1310_64\bin;$env:PATH"

# 新建 Debug 构建目录
mkdir build_debug
cd build_debug

# 用 Qt Kit 生成（按你实际 Qt 版本改路径）
& "D:\Qt\Tools\CMake_64\bin\cmake.exe" `
    -G "MinGW Makefiles" `
    -DCMAKE_BUILD_TYPE=Debug `
    -DCMAKE_PREFIX_PATH="D:\Qt\6.x.x\mingw_64" `
    ..

mingw32-make -j8

方案 B：原地切换

cd build_make_qt
& cmake.exe -DCMAKE_BUILD_TYPE=Debug .
mingw32-make -j8

验证

  • file MotorDev.exe（可用 Git Bash）应显示 not stripped
  • 或者 objdump -h MotorDev.exe | grep debug 能看到 .debug_* 段
  • 体积明显比 Release 大（一般 5–10 倍）
陷阱：如果开了 -O2/-O3，即使有 -g，单步也会出现"行号乱跳、变量显示 <optimized out>"。Debug 构建必须是 -O0 -g（CMake 的 Debug 类型默认就是这个）。

Step 2 配置 VSCode launch.json 10 分钟

装扩展

  • C/C++（Microsoft）—— 提供 cppdbg 调试类型
  • CMake Tools（可选）—— 一键 configure/build

新建 .vscode/launch.json

{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "Debug MotorDev (GDB)",
      "type": "cppdbg",
      "request": "launch",
      "program": "${workspaceFolder}/build_debug/MotorDev.exe",
      "args": [],
      "stopAtEntry": false,
      "cwd": "${workspaceFolder}/build_debug",
      "environment": [
        {
          "name": "PATH",
          "value": "D:\\Qt\\Tools\\mingw1310_64\\bin;D:\\Qt\\6.x.x\\mingw_64\\bin;${env:PATH}"
        },
        {
          "name": "QT_LOGGING_RULES",
          "value": "motordev.*=true"
        }
      ],
      "externalConsole": false,
      "MIMode": "gdb",
      "miDebuggerPath": "D:\\Qt\\Tools\\mingw1310_64\\bin\\gdb.exe",
      "setupCommands": [
        { "description": "Pretty printing", "text": "-enable-pretty-printing", "ignoreFailures": true },
        { "description": "Async mode", "text": "-gdb-set mi-async on", "ignoreFailures": true },
        { "description": "Don't stop on SIGUSR1", "text": "handle SIGUSR1 nostop noprint pass", "ignoreFailures": true }
      ]
    }
  ]
}

关键字段说明

字段作用注意
program调试目标 exe 路径务必指向 Debug 构建
cwd程序的工作目录必须是包含 AW86100.dll 的那个 build 目录，否则 DLL 加载失败
environment.PATH运行时 PATH必须把 Qt bin 和 MinGW bin 加进来，否则启动报"找不到 Qt6Core.dll"

miDebuggerPathGDB 路径Qt 自带的，不要用别处的 GDB（版本不匹配会乱）
mi-async on启用异步模式多线程调试更稳，避免线程切换时卡住
把示例里的 D:\Qt\6.x.x\mingw_64 改成你实际的 Qt 路径。可以在 build_make_qt/CMakeCache.txt 里搜 Qt6_DIR 找到。

Step 3 让 QString / QByteArray 可读 5 分钟

问题

原生 GDB 看 QString 是一坨 QArrayDataPointer<ushort>，看不出实际字符串。Watch 面板里全是噪音。

方案：启用 Qt 的 GDB pretty printers

Qt 自带 Python 脚本，路径一般是：

D:\Qt\6.x.x\mingw_64\share\qtcreator\debugger\qttypes.py
（或在 Qt Creator 安装目录的 share\qtcreator\debugger\）

在 launch.json 的 setupCommands 增加

{
  "description": "Load Qt printers",
  "text": "python exec(open('D:/Qt/Tools/QtCreator/share/qtcreator/debugger/qttypes.py').read())",
  "ignoreFailures": true
}

实在找不到这个文件也没关系，**最低限度**用 GDB 命令查看：

(gdb) p qstring.toUtf8().constData()
(gdb) p (char*)qbytearray.data()

或在 VSCode Watch 面板写：

  • icModel.toUtf8().constData()
  • (char*)payload.data(),payload.size() ← 带逗号长度，可显示二进制内容

Step 4 决定调试源：硬件 vs 模拟器 3 分钟

选项对比

方案能调试到限制
真硬件（STM32 + AW86006） 从 thunk 到 dispatcher 到串口的全链路 需要硬件在手 + 串口连通
调试模拟器（项目自带 SerialDebugTab） thunk → onWriteI2c → syncI2cWrite → dispatcher 这段纯上位机部分 模拟器对 0x30/0x31 透传命令的响应需要确认是否已实现；下载阶段如果模拟器不应答会触发 5 秒超时
注入假 dispatcher（自己写一个） 策略层与协议层，不依赖任何真实串口 需要写一两百行 mock 代码
建议：第一次调试用调试模拟器把上位机部分走通；确认无问题后再接真硬件。

如果用模拟器

  1. 先看 src/services/simulatorservice.cpp 是否处理 CmdI2cTransferWrite (0x30) 和 CmdI2cTransferRead (0x31)
  2. 如果没处理，烧录会卡在第一个 I2C 命令的 5 秒超时
  3. 需要补一段 mock 应答（"无脑成功"即可）才能让流程跑下去

Step 5 战略断点 9 个固定位置 10 分钟

设置原则

不要从入口一路 step——按调用链分层下断点，让程序自动停在你关心的边界，节省精力。

必下的 9 个断点

#位置作用线程
1 fwflashtab.cpp onStartRequested() 烧录入口；确认 IC 名和数据传对了 主线程
2 fwflashservice.cpp runPreflightAndFlash 的 setState(State::Flashing) 那行 worker 即将启动 主线程
3 aw_sdk_strategy.cpp flash() 入口（第 53 行） 已切到 worker 线程 worker
4 doInit / doDownload 各打一个 5 步流程边界 worker
5 extWriteI2cThunk 第一行 DLL 第一次回调你 worker
6 onWriteI2c 入口 thunk 路由到实例 worker
7 syncI2cWrite 第 349 行 QMetaObject::invokeMethod(...) 跨线程提交"前一刻" worker
8 syncI2cWrite 回调 lambda 内 state->received = true 那行（约 339 行） 响应到达，唤醒 worker 主线程（dispatcher 所在）
9 syncI2cWrite 第 363 行 if (state->isError) worker 醒来，准备返回 worker

断点 5 是关键观察点

看到断点 5 命中，确认两件事：

  1. 当前 thread id 与断点 3 一致（同一个 worker）
  2. s_currentInstance 不为空，且指向当前 strategy 实例

这两点验证了 thunk 路由机制工作正常。

断点 8 验证跨线程

断点 8 命中时，VSCode 的 Threads 面板会显示当前在主线程，而 worker 线程仍在断点 7（或它的下一行 cond.wait）等待。这是同一时刻两个线程的快照——直观体现"主线程负责完成，worker 负责等待"。

Step 6 VSCode 调试面板使用要点 8 分钟

必看的 4 个面板

面板用法这功能里关注什么
VARIABLES 展开自动观察当前作用域变量 m_cancelFlag, m_inDownload, m_sentBytes, state
WATCH 固定监视表达式 s_currentInstance, m_cancelFlag->load(), state->received
CALL STACK 当前栈帧；点击切换帧 看清"是谁调到这里"——尤其在 thunk 命中时
THREADS 所有线程列表；点击切换线程焦点 本功能最关键的面板

常用快捷键

动作快捷键对本功能
继续F5主力
Step Over（不进入函数）F10主力
Step Into（进入函数）F11慎用——别 step 进 DLL 或 wait()
Step Out（跳出当前函数）Shift+F11误入 DLL/wait 后救命
切线程点 Threads 面板对应行跨线程接力时必用

条件断点（Conditional Breakpoint）

右键断点 → Edit Breakpoint → 选 Expression。

本功能里几个有用的条件：

位置条件触发场景
onWriteI2c 入口m_cancelFlag->load() == true只在已请求取消时停下
onWriteI2c 入口m_inDownload && wrSize > 32下载阶段大写入
syncI2cWrite 入口m_sentBytes > 1024 * 8跑过 8KB 后才停
提示：对于 onWriteI2c 这种"一次烧录可能调几百次"的回调，无条件断点会让你疯掉。一定要用条件断点或 logpoint。

Step 7 跨线程"接力调试"模板 10 分钟

典型场景：跟踪一次 syncI2cWrite 的全程

步骤

  1. 预先下好断点 6、7、8、9（见 Step 5）
  2. F5 启动，UI 操作触发烧录
  3. 断点 6 命中（worker 线程）—— Threads 面板里有 worker，焦点在 worker
    • 看 VARIABLES：devId / wrSize / wrData 是 DLL 给的入参
  4. F5 继续 → 断点 7 命中（仍在 worker）
    • 看 payload：(char*)payload.data(),payload.size() 在 Watch 里能看二进制
  5. F10 越过 invokeMethod，再 F10 越过 QMutexLocker，到 cond.wait 那一行
    • 不要 F11 进入 wait！wait 内部是平台原语，进去你会迷路且无法回来
    • 在 cond.wait 那一行右键 → Run to Cursor更安全
  6. F5 继续 → 断点 8 命中（主线程）
    • Threads 面板里 worker 显示"运行中"或"等待"，主线程显示当前帧
    • Call Stack 显示从 QMetaObject::invokeMethod 经过事件循环路由过来
  7. F10 越过 cond.wakeOne()，让 worker 醒来
  8. F5 继续 → 断点 9 命中（worker 线程）
    • 切回 Threads 面板看到焦点回到 worker

"接力"的视觉印象

worker:    ─bp6─bp7─[wait]──────────────────[wake]─bp9─→
主线程:                         ─bp8─[wakeOne]─→

每按一次 F5，焦点在两条泳道间跳一次。习惯后会觉得很顺。

易混淆：在断点 8 命中时，worker 线程的状态不是"暂停"，而是"在 wait 里阻塞"——这是程序逻辑，不是调试器干的。F5 让主线程跑完 wakeOne 后，worker 才会真正开始执行下一行。

Step 8 DLL 边界与 QWaitCondition::wait 8 分钟

DLL 没源码，怎么办

  • 不要 F11 进入 m_fnInit / m_fnBootcontrol / m_fnMoveBinDownload / m_fnResetChip——会跳到反汇编视图
  • 正确做法：在调用前一行和调用后一行各下断点 
    // aw_sdk_strategy.cpp doDownload 内
const int ret = m_fnMoveBinDownload(buf, len, ...);   ← BP A 调用前
m_inDownload = false;                                  ← BP B 调用后
  • BP A 和 BP B 之间，DLL 内部会反复调你的 onWriteI2c——你可以F5 跳到 onWriteI2c 的断点看每次写

如果不小心 step 进 DLL

  1. 立刻 Shift+F11 Step Out
  2. 如果还停在汇编，按多次 Shift+F11 直到回到你的源码
  3. 再不行就直接 F5，让它跑完到下一个断点

QWaitCondition::wait 同理

  • F11 进入会进到 Qt 内部、再到 Windows API（WaitForSingleObject），毫无价值
  • 正确做法：在 cond.wait 那一行 F10 越过；在回调 lambda 里下断点观察唤醒
陷阱：进入 DLL 或 wait() 后，VSCode 可能切到反汇编视图（CALL STACK 显示 ??）。**不要慌**，按 Shift+F11 回到你的代码即可。如果完全卡死，重启调试。

Step 9 不打断的诊断手段 8 分钟

1. Logpoint（不暂停的断点）

右键行号 → Add Logpoint，输入消息（用 {表达式} 引用变量）：

onWriteI2c: devId=0x{devId,X} size={wrSize} sent={m_sentBytes}

命中时不停，只把消息打到 DEBUG CONSOLE。适合 onWriteI2c 这种高频回调。

2. qDebug + 线程 ID

临时往代码里塞：

qDebug() << "[I2C-W]"
         << "tid=" << QThread::currentThreadId()
         << "size=" << wrSize
         << "sent=" << m_sentBytes;

对验证"哪一段在哪个线程"非常有效。验证完务必删掉。

3. 启用项目 Q_LOGGING_CATEGORY

项目里 fwflashservice.cpp:23 已经定义了 motordev.fwflash 类别。launch.json 里那行 QT_LOGGING_RULES=motordev.*=true 就是为了启用它。运行时直接在控制台看：

motordev.fwflash: 步骤 1/5：AwOIS_ExtFuncInit
motordev.fwflash: 已加载 DLL：AW86100.dll
...

4. GDB 命令行（高级）

VSCode 的 DEBUG CONSOLE 支持直接打 GDB 命令，前缀 -exec：

-exec info threads               # 查看所有线程
-exec thread 3                   # 切到线程 3
-exec bt                         # 打印当前栈
-exec print s_currentInstance    # 看静态变量
-exec watch m_sentBytes          # 数据断点：变化即停

watch 数据断点对调试"为什么 m_sentBytes 突然不增长了"这类问题特别有用——它的变化时机就是停下来的时机。

Step 10 常见症状速查表 参考用

症状原因对策
启动报"找不到 Qt6Core.dll" launch.json 的 PATH 没把 Qt bin 加进来 检查 environment.PATH
启动后立即崩溃 / 报缺 AW86100.dll cwd 不对，DLL 不在工作目录 把 cwd 改成包含 DLL 的目录
断点显示 ⭕ 灰圆"Unverified" 构建不是 Debug，或者源文件路径不匹配 重建 Debug，确认 launch.json 的 program 指向新 exe
变量显示 <optimized out> 带了 -O2 优化 确认 CMAKE_BUILD_TYPE=Debug，重建
F10 后行号乱跳 同上：优化导致行号映射错乱 同上
断点不命中，但代码确实跑了 路径区分大小写问题，或者改了源文件没重新编译 重新构建；或在 DEBUG CONSOLE -exec info b 检查 BP 实际地址
thunk 命中，但 s_currentInstance 是 0 thread_local 初始化时机问题，或不是同一个线程 在 flash() 入口确认 s_currentInstance = this 已执行；对比 thread id
cond.wait 后再也不醒 响应没回来（超时 5s 仍没动） 确认串口/模拟器是否回了响应；检查 dispatcher 的 seq 匹配
每次都卡在第一个 I2C 命令 5s 超时 用模拟器调试但模拟器没处理 0x30/0x31 给 SimulatorService 加 0x30/0x31 的 mock 应答
VARIABLES 里 QString 看不到内容 没启用 Qt pretty printers Step 3 的 setupCommands；或用 .toUtf8().constData()
调试中点取消按钮没反应 worker 在 cond.wait 等待中，要等到响应或超时才检查 cancelFlag 这是设计行为，不是 bug；最坏 5s 后会响应取消
