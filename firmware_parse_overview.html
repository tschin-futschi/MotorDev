固件文件选择 — .bin 与 .hex 解析流程

项目：MotorDev  |  范围：src/protocol/firmware_parser.{h,cpp} · src/tabs/fwflashtab.cpp 中的 parseAndShowFile()

用户点击的是「浏览…」按钮（项目里没有"上传"这个按钮，只有选择文件 + 后续点"开始"才真正烧录）。

一、流程入口与总链路

onBrowseClicked() → QFileDialog 拿路径 → parseAndShowFile(path)
                                              ↓
                          FirmwareParser::parseFile(path)
                                              ↓
                  公共预检 → 按后缀分流 → parseBin / parseHex

二、公共预检（两种格式都走）

位置：firmware_parser.cpp:223-255 的 FirmwareParser::parseFile()

  1. QFile::open(ReadOnly) 打开文件，失败直接报错
  2. 取文件大小，若 <= 0 → "文件为空"
  3. 若 size > 1024 KB → 超限报错
  4. f.readAll() 一次性读入 QByteArray content
  5. 取 fi.suffix().toLower()：
    • "bin" → parseBin()
    • "hex" → parseHex()
    • 其它 → "不支持的文件格式"

三、.bin 文件路径（极简）

位置：firmware_parser.cpp:53-63 的 parseBin()

info.fileName       = fi.fileName();
info.fileSizeBytes  = content.size();
info.format         = FirmwareFormat::Bin;
info.data           = content;                    // 直接赋值，零拷贝（隐式共享）
info.effectiveBytes = content.size();
info.crc32          = computeCrc32(content);
info.valid          = true;

就五件事：

  • 复制元信息（文件名、大小）
  • data 字段直接挂上原始字节——.bin 本身就是连续映像，无需任何转换
  • 算 CRC32（IEEE 802.3 标准，poly 0xEDB88320）
  • 标 valid = true
  • segments / minAddress / maxAddress 保持默认（空 vector，0/0），UI 信息面板对 .bin 不会显示这些字段

效率上非常便宜：除一次 CRC32 遍历外没有任何额外内存开销。

四、.hex 文件路径（重活全在这里）

位置：firmware_parser.cpp:71-215 的 parseHex()，是一套完整的 Intel HEX 解析器。

1. 行级解析（逐行扫描整个文本）

每一行格式：:LL AAAA TT DD...DD CC

  • : 起始符——缺则报错
  • LL：1 字节数据长度（byteCount）
  • AAAA：2 字节地址低位
  • TT：1 字节记录类型
  • DD...DD：byteCount 字节数据
  • CC：1 字节校验和

每行检查 4 项：

  1. 起始符 :
  2. 长度合法性（最少 10 字符 + 偶数）
  3. 行长度必须等于 10 + byteCount * 2
  4. 校验和：(-(LL + AA + AA + TT + Σdata)) & 0xFF 必须等于 CC，否则报错

2. 记录类型分支

类型名称处理
0x00Data把 byteCount 个字节按绝对地址写入 QMap<quint32, quint8>，地址 = upperAddr | addr16
0x01EOF标记 eofSeen = true
0x04Extended Linear Address更新 upperAddr = (hb<<24) | (lb<<16)，给后续 Data 行用作高位地址
0x05Start Linear Address忽略（启动入口地址不影响烧录内容）
0x02 / 0x03Segment Addr 类直接报错（暂不支持）
其它未知报错

3. 收尾校验

  • 若没看到 0x01 EOF 行 → 报错"缺少 EOF 记录"
  • 若 bytes 为空 → 报错"文件不包含任何数据记录"

4. 段合并 + 0xFF 填充（最关键的"重活"）

QMap 自动按 key 排序。遍历键序列：

  • 若当前地址 == 上一地址 + 1 → 同一段延续
  • 若有跳变 → 把上一段 {segStart, length} 推入 info.segments，开新段

收完后还会：

info.minAddress = QMap 第一个 key
info.maxAddress = QMap 最后一个 key
const quint64 totalLen = maxAddress - minAddress + 1;

并再做一次大小检查：totalLen > 1024 KB 也报错。

接着真正干"合并"：

info.data = QByteArray(totalLen, '\xFF');                 // 整段填 0xFF
for (auto kt = bytes.begin(); kt != bytes.end(); ++kt) {
    info.data[kt.key() - minAddress] = kt.value();        // 把每个有数据的字节覆盖回去
}
info.crc32 = computeCrc32(info.data);
info.valid = true;

5. 最终结果对比

字段.bin.hex
formatBinHex
data原始字节minAddress→maxAddress 连续映像，空隙填 0xFF
effectiveBytes= fileSizeBytes实际有数据的字节数（QMap 大小）

五、解析完成后 UI 怎么用结果

回到 FwFlashTab::parseAndShowFile()（fwflashtab.cpp:318-341）：

  1. m_infoPanel->setInfo(info) —— 让 FwFileInfoPanel 切换到对应页面
    • valid == false → 错误页（显示 errorMessage）
    • valid == true → 合法页（显示文件名、大小、CRC32；若是 hex 还显示 segments 列表与地址范围）
  2. valid == true 时 m_currentFirmwareData = info.data —— 这就是后面要烧的字节
  3. m_logPanel->appendInfo(...) —— 控制台输出 已加载固件：xxx (N 字节，CRC32 0x........)
  4. updateStartEnabled() —— 满足"IC 已选 + 文件 valid + service 不忙"才点亮"开始"按钮
到这里只是解析 + 暂存，串口上还没发任何字节。真正烧录要等用户点"开始"，触发 onStartRequested() → m_service->startFlash(icModel, m_currentFirmwareData, m_currentFirmwareTotal)。

六、核心差异一句话总结

  • .bin：原样收下，算个 CRC32 就走人。
  • .hex：解码 ASCII → 校验每行 → 收集每个绝对地址字节 → 用 0xFF 把空隙填成连续映像 → 再算 CRC32。

对下游策略 flash() 来说两者完全等价：拿到的都是一段连续 QByteArray，无需关心来源。这就是为什么策略接口签名只接受 firmware，没有 format 参数。
