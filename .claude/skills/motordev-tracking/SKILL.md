# MotorDev 问题记录

## 概述

使用本技能将已经明确的问题、阻塞、结论或调试摘要，整理成写给用户阅读的 HTML 记录文件，保存到 `TRACKING/`。

本技能是窄技能，只负责 `TRACKING` 记录，不替代 `CLAUDE.md`、`tmp_handoff/` 或正常实现流程。

## 触发条件

当用户提出以下意图时使用本技能：

- 总结问题
- 记录问题
- 归档问题
- 记录阻塞
- 保留调试结论
- 生成便于后续查看的用户可读记录

以下情况不要使用本技能：

- 需要创建阶段计划时，应写入 `tmp_handoff/plan_*.md`
- 需要提交实现回执时，应写入 `tmp_handoff/report_*.md`
- 需要给出 Review 结论时，应写入 `tmp_handoff/review_*.md`
- 正常编码、Review、调试分析时，不应默认读取 `TRACKING/`

## 输出位置与命名

- 输出目录固定为：`TRACKING/`
- 输出文件格式固定为：`.html`
- 文件名必须包含：
  - 问题主描述
  - 时间戳

建议命名格式：

`<problem-summary>_<yyyymmdd>_<hhmmss>.html`

其中：

- `problem-summary` 要短、可读、适合作为文件名
- 避免使用空格、过长标题和不安全字符
- 时间戳用于区分不同记录版本

## 记录内容结构

面向用户阅读，通常包含以下内容：

- 标题
- 记录时间
- 当前状态
- 问题摘要
- 现象描述
- 影响范围或风险
- 已做检查 / 已知证据
- 当前判断或原因假设
- 后续动作 / 待确认项

如果信息不足，不要硬凑；明确写出未知项。

## 写作规则

- 以中文记录
- 面向用户，不面向另一个 LLM
- 内容简洁、可扫描
- 事实与推测分开写
- 明确保留重要文件名、类名、命令、报错信息和结论
- 不要把整段聊天记录原样贴进去
- 不要用空泛总结替代具体事实
- 不要生成"看起来完整但没有信息量"的套话

## 上下文规则

- `TRACKING/` 中的文件是用户可读记录，不是默认工作上下文
- 后续任务中，只有在用户明确引用某条记录，或当前问题与该记录直接相关时，才去读取它
- 正常实现、Review、阶段交接时，不要默认扫描 `TRACKING/`

## 与其他项目文件的关系

- `CLAUDE.md`：项目总规则与技术负责人规则
- `tmp_handoff/`：阶段计划、实现回执、Review 结论
- `TRACKING/`：面向用户的问题追踪记录

不要混用这三类文件的职责。
