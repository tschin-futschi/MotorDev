0. 读代码前的 C++ 速览

下面这 5 个概念会反复出现，先简单认识，后面遇到再展开：

0.1 头文件 (.h) 与源文件 (.cpp)

C++ 把代码分成两类文件：

  • .h（头文件，header）— 写"声明"。告诉别人"这里有一个类，它有这些成员函数、签名是什么"。其他文件用 #include "xxx.h" 把声明拉进来
  • .cpp（源文件，implementation）— 写"实现"。函数体里具体做什么

类比：头文件像目录与函数原型清单，源文件像每个函数的具体内容。读代码时先看 .h 知道有什么，再看 .cpp 知道怎么做。

0.2 类（class）的基本结构

class MyClass {
public:
    int doSomething();   // 外人能调用
private:
    int m_count;          // 外人看不见，类内部用
};
  • public: 后面声明的东西外部代码能访问
  • private: 后面声明的只能在类内部用
  • protected: 介于二者之间：本类和子类能用，外人不行
  • 本项目约定：成员变量用 m_ 前缀

0.3 指针（*）和引用（&）

这两个东西在本项目代码里到处都是。简单理解：

  • 值传递：void f(int x) — 函数收到 x 的副本，改 x 不影响外面
  • 指针传递：void f(int *x) — 函数收到的是地址，*x = 5 会改外面的值；x 可以是 nullptr（空）
  • 引用传递：void f(int &x) — 函数收到的也是地址，但语法上当变量用，x = 5 会改外面的值；不能为空
  • 常量引用：void f(const int &x) — 引用，但函数不能改它。最常用，传大对象不复制又安全

0.4 命名空间 (namespace)

std::function、std::atomic 这种 :: 形式叫"作用域解析"。std::xxx 表示来自 C++ 标准库的 xxx。Qt 库的对象（QString、QObject）一般不需要前缀。

0.5 Qt 几个常见类型

类型作用
QStringQt 的字符串类，比 std::string 更方便处理 Unicode
QByteArray字节数组，可以放任意二进制（比如固件数据）
QObjectQt 所有"有信号槽功能"的类的基类
Q_OBJECT放在类声明里的宏，让 Qt 元对象系统能识别这个类

1. flashstrategy.h — 烧录的"接口契约"

这个文件 40 行，是整个烧录功能的"地基"。它告诉所有人："本项目里所谓'烧录'，就是实现下面这个接口"。

📄 src/services/flashstrategy.h（完整代码）
// =============================================================================
// @file    flashstrategy.h
// @brief   烧录策略接口 — 不同 IC 的烧录算法抽象基类
// =============================================================================
#pragma once

#include <QByteArray>
#include <QString>

#include <atomic>
#include <functional>

class FlashStrategy {
public:
    virtual ~FlashStrategy() = default;

    /// @brief 下拉框显示的 IC 型号文本
    virtual QString icModel() const = 0;

    /// @brief IC 描述（次要文字，可选）
    virtual QString icDescription() const { return {}; }

    /// @brief 烧录入口，由 FwFlashService 在 worker 线程调用
    virtual bool flash(const QByteArray &firmware,
                       std::function<void(qint64 sentBytes)> progress,
                       const std::atomic<bool> &cancelFlag,
                       QString *errorMessage) = 0;
};

1.1 文件开头：#pragma once 与 #include

#pragma once

#include <QByteArray>
#include <QString>
#include <atomic>
#include <functional>
#pragma once 是头文件防重复包含的指令。同一个头文件可能被多个 .cpp 间接包含，没有这一行就会编译报"重复定义"。每个 .h 文件都该写。

#include <X> 是把另一个头文件的内容"贴"到当前位置。<...> 是系统/库头文件，"..." 是项目自己的头文件。

这 4 个 include 引入了：Qt 的 QByteArray（字节数组）、QString（字符串）、C++ 标准库的 std::atomic（原子类型）、std::function（函数对象）。后面代码用到时就不用再解释这些类型从哪来。

1.2 类声明的开头

class FlashStrategy {
public:
    virtual ~FlashStrategy() = default;

class FlashStrategy { ... } 声明一个名叫 FlashStrategy 的类。public: 表示下面声明的成员对外可见。

~FlashStrategy() 是析构函数（destructor）— 类名前加个 ~。当对象被销毁时这个函数会自动调用，用来清理资源。= default 让编译器自动生成默认实现（什么都不做）。

关键字 virtual — 告诉编译器："这个函数子类可以重写（override）"。

为什么析构函数要写 virtual？因为 FlashStrategy 是基类，会被子类继承。当代码通过基类指针删除对象时（比如 delete strategyPtr），如果析构函数不是 virtual，只会调基类的析构，子类那部分资源不会被清理 — 这是 C++ 经典内存泄漏陷阱。所以"有虚函数的类，析构函数必须也是虚的"是 C++ 铁律。

1.3 三个核心虚函数

    virtual QString icModel() const = 0;

    virtual QString icDescription() const { return {}; }

    virtual bool flash(...) = 0;
虚函数 (virtual function)：带 virtual 关键字的成员函数。它的特点是"调用时看实际对象类型，不看指针类型"。

比如：基类 Animal 有虚函数 speak()，子类 Dog 重写为 "汪"，Cat 重写为 "喵"。代码 Animal *p = new Dog(); p->speak(); 会输出 "汪" — 即使指针类型是 Animal*。这就是多态。

纯虚函数 (pure virtual)：函数声明后面加 = 0。意思是"基类没有实现，子类必须重写"。一个类只要有一个纯虚函数，就叫抽象类，不能直接实例化。

函数末尾的 const：放在 () 后面，表示"这个函数不会修改对象的成员变量"。读取数据的函数都该加 const，是 C++ 良好习惯。

所以这三行的意思是：

  • icModel() — 纯虚函数，子类必须实现，返回 IC 型号字串（比如 "AW86006"）
  • icDescription() — 普通虚函数，有默认实现（返回空 QString），子类可以选择不重写
  • flash() — 纯虚函数，子类必须实现真正的烧录算法
为什么用虚函数？因为本项目要支持多款 IC（AW86006 / AW86100 / DW9786 / DW9788），每款 IC 烧录算法可能不同。我们希望写出这样的代码：
FlashStrategy *strategy = findStrategyByName("AW86006");
strategy->flash(firmware, ...);   // 不管是哪个 IC，调法都一样
这就是多态。如果不用虚函数，每加一款 IC 就要改一堆 if (icModel == "AW86006") ... else if (...)，代码会乱掉。虚函数让"调用方"和"具体实现"解耦。

1.4 flash() 的 4 个参数详解

    virtual bool flash(const QByteArray &firmware,                          // ①
                       std::function<void(qint64 sentBytes)> progress, // ②
                       const std::atomic<bool> &cancelFlag,            // ③
                       QString *errorMessage) = 0;                       // ④

① const QByteArray &firmware — 输入：固件字节流

常量引用方式传 QByteArray。引用避免复制大数据；const 保证函数不会改它（只读输入）。

② std::function<void(qint64 sentBytes)> progress — 进度回调

std::function 是 C++ 标准库提供的"函数对象容器"。它能装下任何"调用方式与签名匹配的东西" — 普通函数、lambda、成员函数等。

std::function<void(qint64)> 的意思是："一个能用 qint64 参数调用、返回 void 的可调用对象"。

类比 JavaScript：就是把一个 callback 函数作为参数传进来，函数内部到时候调用它。

所以这个参数的意思是："你（调用方）给我一个回调，我每发送一段数据就调一次，告诉你已经发了多少字节"。子类实现 flash() 时会反复调 progress(已发字节数)，UI 收到后更新进度条。

③ const std::atomic<bool> &cancelFlag — 取消标志

std::atomic<T> 是 C++ 的"原子类型"模板。它保证读写操作是不可分割的（atomic），多线程同时访问不会出错。

普通的 bool 在多线程里读写有数据竞争（race condition）的风险；std::atomic<bool> 没有。本项目里这个标志由主线程写（用户点取消时置 true）、worker 线程读，所以必须是 atomic 的。

注意是 const 引用 — 烧录函数不能修改这个标志（修改的事归 Service / UI），只能读它。

④ QString *errorMessage — 出参：失败原因

用指针传出错信息。函数失败时往 *errorMessage 里写错误描述；成功时不写。

这里用指针而不是引用，是为了让"不需要错误信息时"调用方能传 nullptr。引用不能为空，指针可以。在 C++ 风格里，"可选的输出参数"通常用指针。

1.5 注释里的 4 条调用约定

头文件类注释里写了：

// flash() 调用约定：
// - 同步阻塞，运行在 worker 线程
// - 通过 progress 回调上报已发送字节数（不必每字节，按合理粒度即可）
// - cancelFlag 由外部翻转为 true 时应在下一安全点退出并返回 false
// - 失败时填充 errorMessage（成功时不写）

这是子类实现 flash() 时必须遵守的契约。后面看 AwSdkStrategy::flash() 时你会看到这 4 条都被认真实现了。

1.6 第一个文件总结

读完这一个文件你应该明白：

  1. "烧录"在本项目里被抽象成一个虚函数 flash()
  2. 每款 IC 写一个继承 FlashStrategy 的子类，实现自己的 flash()
  3. 调用方通过基类指针调 flash()，C++ 多态自动找到对应子类的实现

2. aw_sdk_strategy.h — 真正烧录算法的声明

这个文件 130 多行，比上一个复杂得多。我们只挑关键的几段讲。完整文件你打开 IDE 自己对照。

2.1 类声明：继承

📄 src/services/flashstrategies/aw_sdk_strategy.h（节选）
class AwSdkStrategy : public FlashStrategy {
public:
    enum class LogLevel { Info, Warn, Error, Ok };
    using LogSink = std::function<void(LogLevel level, const QString &message)>;

    AwSdkStrategy(CommandDispatcher *dispatcher, LogSink logSink);
    ~AwSdkStrategy() override;

    bool flash(const QByteArray &firmware,
               std::function<void(qint64)> progress,
               const std::atomic<bool> &cancelFlag,
               QString *errorMessage) override;
继承（inheritance）：class A : public B 表示 A 继承 B。A 自动拥有 B 的所有成员（除了构造/析构），还可以新增自己的成员，或重写 B 的虚函数。

public 继承：意味着"A 是 B 的一种"（is-a）。本项目里 AwSdkStrategy 继承 FlashStrategy，意思是"AwSdkStrategy 是一种烧录策略"。

override 关键字：放在子类函数末尾，表示"我重写了基类的虚函数"。如果你写错了签名（比如参数类型不对），编译器会报错，避免你以为重写了实际没重写的尴尬 bug。

2.2 enum class 与 using 别名

    enum class LogLevel { Info, Warn, Error, Ok };
    using LogSink = std::function<void(LogLevel level, const QString &message)>;
enum class 是"强类型枚举"。比传统的 enum 更安全 — 不会和数字隐式互转，使用时必须带名字：LogLevel::Info 而不是 Info。

using X = Y 是"类型别名"，给一个长类型起个短名字。比如这里 LogSink 就是 std::function<void(LogLevel, const QString&)> 的别名，后面写 LogSink 比写一长串顺眼得多。

（旧式 C 写法是 typedef，新式 C++ 推荐 using。）

2.3 构造函数

    AwSdkStrategy(CommandDispatcher *dispatcher, LogSink logSink);

构造函数是与类同名的特殊函数，对象创建时自动调用。这个构造函数接收两个参数：

  • CommandDispatcher *dispatcher — 串口命令分发器的指针，烧录时用来发协议帧
  • LogSink logSink — 一个回调函数对象，用于把日志送到 UI 面板

这两个东西是外部注入的依赖。本类自己不关心 dispatcher 来自哪里、日志最终去哪里，只管按接口调用即可。这就是依赖注入设计模式。

2.4 私有部分：DLL 接口的复刻

private:
    using DllInt    = int;
    using DllByte   = unsigned char;
    using DllUInt32 = unsigned int;

    using FnReadI2c   = DllInt (*)(DllByte devId, DllByte addrSize, DllByte *pAddr,
                                    DllByte rdSize, DllByte *pRdBuf);
    using FnWriteI2c  = DllInt (*)(DllByte devId, DllByte wrSize, DllByte *wrData);
    using FnOutputLog = DllInt (*)(const char *str);
函数指针类型：DllInt (*)(参数...) 这种语法定义的是"指向函数的指针"。

比如 FnReadI2c 是"一个指向某函数的指针类型，那个函数返回 int、参数是 5 个"。这样后面就可以声明变量 FnReadI2c m_fnRead; 把它指向某个函数。

为什么要复刻？因为厂家 SDK 头文件 AW86100.h 里有这些函数指针类型的定义，但本项目故意不 include 它，原因是希望联调时能用 dummy DLL 替换真实 DLL 而不影响代码编译。所以在这里用 using 复刻一份等价的类型定义。

2.5 静态 thunk 函数与 thread_local 实例指针

    static int extReadI2cThunk(DllByte devId, DllByte addrSize, DllByte *pAddr,
                                DllByte rdSize, DllByte *pRdBuf);
    static int extWriteI2cThunk(DllByte devId, DllByte wrSize, DllByte *wrData);
    static int extOutputLogThunk(const char *str);

    // thunk 转发到的实例方法
    int onReadI2c(DllByte devId, DllByte addrSize, DllByte *pAddr,
                   DllByte rdSize, DllByte *pRdBuf);
    int onWriteI2c(DllByte devId, DllByte wrSize, DllByte *wrData);
    int onOutputLog(const char *str);

    // 静态 thread_local 实例指针（C-style 回调路由）
    static thread_local AwSdkStrategy *s_currentInstance;
静态成员函数（static）：不属于具体对象，属于"类本身"。调用时不需要对象，也没有 this 指针，没法直接访问对象的成员变量。语法上像普通函数。

实例成员函数：属于对象，调用时有隐式 this 指针，能访问 m_xxx 成员变量。

thread_local 关键字：每个线程独立持有一份这个变量。线程 A 改了不会影响线程 B 看到的值。
为什么需要 thunk + thread_local？

厂家 DLL 是 C 接口，它收到的 3 个回调必须是普通 C 函数指针（或等价的静态函数）。但成员函数有隐式的 this 参数，签名根本对不上 — 没法直接传给 DLL。

解决方案：
① 给 DLL 传一个静态函数 extWriteI2cThunk（叫它"thunk"，垫片）
② thunk 内部从 s_currentInstance 这个全局变量找到"当前正在烧录的实例"
③ thunk 调用 s_currentInstance->onWriteI2c(...) 转发到实例方法
④ 之所以是 thread_local，是为了允许多个线程各自跑各自的烧录而互不干扰

这是 C++ 程序对接 C 库的经典套路，看到这个模式不要慌。

2.6 第二个文件小结

读完这个头文件你应该明白：

  • AwSdkStrategy 是 FlashStrategy 的子类，重写了 flash()
  • 它内部要加载一个 DLL，DLL 通过 3 个 C 风格的函数指针回调本类
  • 因为 C 接口和 C++ 成员函数不兼容，用了 thunk + thread_local 模式做转接

3. aw_sdk_strategy.cpp::flash() — 5 步流程主干

真正的烧录算法。我们逐段看。

📄 src/services/flashstrategies/aw_sdk_strategy.cpp（flash 函数）
bool AwSdkStrategy::flash(const QByteArray &firmware,
                          std::function<void(qint64 sentBytes)> progress,
                          const std::atomic<bool> &cancelFlag,
                          QString *errorMessage) {
    // 状态复位
    m_cancelFlag = &cancelFlag;
    m_progress = std::move(progress);
    m_sentBytes = 0;
    m_initCalled = false;
    m_bootEntered = false;

3.1 函数签名与状态复位

bool AwSdkStrategy::flash(...) — AwSdkStrategy:: 表示"这是 AwSdkStrategy 类的成员函数"。这是头文件声明的同名函数的实现。

函数体一开始把对象成员变量复位到初始值：

  • m_cancelFlag = &cancelFlag; — &cancelFlag 是"取地址"，把引用变成指针，存起来给后面用
  • m_progress = std::move(progress); — 把 progress 的内容"移动"到成员变量。std::move 是性能优化，避免复制
  • s_currentInstance = this; — 关键：把当前对象设为"当前实例"，让 thunk 能找到
this 指针：在成员函数内部，this 是个隐藏参数，指向"调用本函数的那个对象"。m_xxx = ... 等价于 this->m_xxx = ...。

std::move：标记一个对象"我不再需要它的内容了，你可以拿走"。配合"移动构造函数"实现零拷贝转移。第一遍读 std::move(x) 当成 x 即可。

3.2 两个辅助 lambda

    auto bail = [&](bool ok) -> bool {
        if (!ok) emergencyCleanup();
        s_currentInstance = nullptr;
        m_cancelFlag = nullptr;
        m_progress = nullptr;
        return ok;
    };

    auto checkCancel = [&]() -> bool {
        if (m_cancelFlag != nullptr && m_cancelFlag->load()) {
            if (errorMessage) *errorMessage = QStringLiteral("用户取消");
            return true;
        }
        return false;
    };
Lambda（匿名函数）：语法是 [捕获列表](参数) -> 返回类型 { 函数体 }。它就是一个"就地写出来的小函数"。

捕获列表 [&]：表示"按引用捕获外面所有用到的变量"。这样 lambda 内部能访问外面函数的局部变量、参数。

其他常见捕获：[=] 按值捕获、[x] 只捕获 x、[&x, y] 引用捕获 x、值捕获 y。

auto：让编译器自动推导类型。Lambda 的真实类型很复杂，用 auto 最方便。

这两个 lambda 的作用：

  • bail(ok) — 函数退出时调用。如果失败就跑 emergencyCleanup()（强制收尾），然后清空所有状态变量，最后返回 ok。
  • checkCancel() — 检查取消标志。如果用户取消了，写错误信息并返回 true。

m_cancelFlag->load() — -> 是"指针访问成员"。load() 是 std::atomic 的方法，原子地读取当前值。

3.3 5 步流程主体

整段都是同一个套路反复出现：

  1. 检查取消标志 — 用户取消了就立即退出
  2. 调一个 doXxx() 函数（内部调 DLL 对应的导出函数）
  3. 失败就调 bail(false) 走收尾路径并返回 false

5 步顺序是厂家 SDK 死规定：

步骤DLL 函数意义
0QLibrary::load()把 AW86100.dll 加载到进程
1AwOIS_ExtFuncInit把 3 个 thunk 函数指针注入 DLL
2AwBootcontrol把 IC 切到 boot 模式
3AwMoveBinDownload真正下载，DLL 内部反复调写 I2C 回调
4AwResetChip复位 IC，让它退出 boot 回到正常运行
5Aw_OIS_ExtFuncDeInit清理 DLL 内部状态

4. aw_sdk_strategy.cpp::onWriteI2c & syncI2cWrite — 数据怎么发出去的

上一节的 doDownload() 调 DLL 的 AwMoveBinDownload，DLL 在内部把固件切片，每片调一次写 I2C 回调。回调是 thunk → 实例方法的二段式。我们看实例方法：

4.1 thunk 转发到实例方法

int AwSdkStrategy::extWriteI2cThunk(DllByte devId, DllByte wrSize, DllByte *wrData) {
    if (s_currentInstance == nullptr) return -1;
    return s_currentInstance->onWriteI2c(devId, wrSize, wrData);
}

这就是 thunk 的全部内容：从 s_currentInstance 拿到当前对象，调它的 onWriteI2c()。一行干活。

4.2 onWriteI2c — 取消检查 + 解析 + 同步发送

int AwSdkStrategy::onWriteI2c(DllByte devId, DllByte wrSize, DllByte *wrData) {
    // ① 取消传播：返回非零让 DLL 自己中止
    if (m_cancelFlag && m_cancelFlag->load()) return -2;

    // ② 解析 wrData：第一字节是寄存器地址长度，后面是地址 + 数据
    uint8_t addrSize = wrData[0];
    const uint8_t *addr = &wrData[1];
    uint8_t dataLen = wrSize - 1 - addrSize;
    const uint8_t *data = wrData + 1 + addrSize;

    // ③ 同步发送 I2C 写命令
    int rc = syncI2cWrite(devId, addrSize, addr, dataLen, data, kI2cTimeoutMs);

    // ④ 累计字节数驱动进度条
    if (m_inDownload) notifyProgress(dataLen);

    return rc;
}

4 步：检查取消 → 解析 DLL 给的字节流 → 调 syncI2cWrite 真发送 → 累计进度。

取消传播为什么是"返回非零"而不是"直接 break"？
因为 onWriteI2c 是 DLL 通过函数指针回调的，控制权在 DLL 手里。我们没法"直接退出"，只能通过返回值告诉 DLL："这次写失败了"。DLL 收到非零返回会自动中止下载循环并返回错误，控制权这才回到我们的 doDownload()。

4.3 syncI2cWrite — 把异步包装成同步

这是整段代码最有技术含量的地方。简化版代码：

int AwSdkStrategy::syncI2cWrite(uint8_t devId, ..., int timeoutMs) {
    QMutex mutex;
    QWaitCondition cond;
    bool done = false;
    int result = 0;

    // ① 把请求发到主线程的 dispatcher（worker 线程不能直接调）
    QMetaObject::invokeMethod(m_dispatcher, [&]() {
        m_dispatcher->sendI2cWrite(devId, addr, data, [&](int rc) {
            // ③ 主线程收到响应后调这个回调
            QMutexLocker lock(&mutex);
            result = rc;
            done = true;
            cond.wakeAll();
        });
    }, Qt::QueuedConnection);

    // ② worker 线程在这里阻塞等待响应
    QMutexLocker lock(&mutex);
    if (!done) cond.wait(&mutex, timeoutMs);
    return done ? result : -3;
}
QMutex 是互斥锁，多个线程访问同一资源时用它防止冲突。QMutexLocker lock(&mutex) 是 RAII 风格的自动加锁/解锁 — 进作用域加锁，出作用域自动解锁，省得手动 unlock()。

QWaitCondition 是条件变量。一个线程可以在它上面 wait() 阻塞，另一个线程调 wakeAll() 唤醒它。这是经典的"生产者-消费者"同步原语。

QMetaObject::invokeMethod(obj, lambda, Qt::QueuedConnection) 把 lambda "投递"到 obj 所在线程的事件队列里执行。这是 Qt 跨线程调用的标准写法。
整段为什么这么写？

问题：worker 线程要发 I2C 写命令，但 CommandDispatcher 在主线程，SerialManager 又在另一个独立线程。worker 不能直接操作它们（线程安全问题）。同时 DLL 要求"写 I2C 必须同步返回结果"，所以又不能简单地"发出去就走人"。

解法：
① worker 通过 invokeMethod 把"发 I2C"这件事排到主线程
② worker 自己用 QWaitCondition 阻塞睡觉
③ 主线程 dispatcher 异步处理，发完串口、收到响应后回调里 wakeAll() 把 worker 叫醒
④ worker 醒来拿到结果返回给 DLL

这样：worker 看到的是"同步阻塞"的接口（合 DLL 心意），实际底层走的是异步事件循环（合 Qt 多线程模型）。异步包同步是 Qt 多线程编程的经典套路。
这段代码涉及 3 个线程跳跃：worker → 主线程 dispatcher → SerialManager 串口线程 → 响应回主线程 → 唤醒 worker。第一遍读不需要把每个线程都映射清楚，记住"worker 这里阻塞了几毫秒到几秒，等响应回来才继续"就够了。

5. fwflashservice.cpp — flash() 是怎么被启动的

到目前为止我们看的都是"算法本身"。但谁来调 flash()？谁创建 worker 线程？这就是 FwFlashService 的活。

5.1 startFlash 入口

📄 src/services/fwflashservice.cpp（startFlash 简化）
void FwFlashService::startFlash(const QString &icModel,
                                const QByteArray &firmware, qint64 totalBytes) {
    if (isBusy()) return;                            // 防重
    FlashStrategy *strategy = m_registry->find(icModel);  // 按 IC 型号查策略
    if (strategy == nullptr) return;
    if (firmware.isEmpty()) return;

    m_totalBytes = totalBytes;
    m_cancelFlag = std::make_shared<std::atomic<bool>>(false);  // 创建取消标志

    setState(State::Preparing);
    runPreflightAndFlash(strategy, firmware);                       // 进入主流程
}

这一段是校验和准备。m_registry->find(icModel) 是关键 — 通过策略名字找到对应的 AwSdkStrategy 实例（FlashStrategy * 类型，多态调用）。

std::shared_ptr<T> 是"共享所有权智能指针"。多个地方可以同时持有同一个对象的 shared_ptr，每多一份引用计数 +1，每销毁一份 -1，计数归 0 时自动释放对象。

std::make_shared<T>(args...) 是创建 shared_ptr 的标准方法。比直接用 new 更高效安全。

本项目里 m_cancelFlag 用 shared_ptr<atomic<bool>> 而不是普通成员，是为了保证 worker 线程访问它时不会野指针（即使 Service 先销毁，worker 持有的 shared_ptr 仍然有效）。

5.2 创建 worker 线程

void FwFlashService::runPreflightAndFlash(FlashStrategy *strategy, const QByteArray &firmware) {
    // ... 前面省略 4 步停止序列 ...

    setState(State::Flashing);

    auto cancelFlag = m_cancelFlag;        // shared_ptr 拷贝（引用计数 +1）
    const qint64 total = m_totalBytes;
    QPointer<FwFlashService> self(this);  // 弱指针，防 service 提前销毁

    m_workerThread = QThread::create([self, strategy, firmware, cancelFlag, total]() {
        QString errorMsg;
        auto progress = [self, total](qint64 sent) {
            if (self.isNull()) return;
            emit self->progressUpdated(sent, total);   // 跨线程信号
        };

        // 真正的烧录调用！worker 线程在这里被阻塞几秒到十几秒
        const bool ok = strategy->flash(firmware, progress, *cancelFlag, &errorMsg);

        // 完成后把结果送回主线程切状态
        QMetaObject::invokeMethod(self.data(), [self, ok, errorMsg]() {
            if (ok) self->setState(State::Completed);
            else    self->setState(State::Failed);
            emit self->finished(ok, ok ? "成功" : errorMsg);
        }, Qt::QueuedConnection);
    });
    m_workerThread->start();
}
QPointer<T> 是 Qt 的"弱指针"，只能指向 QObject 子类。被指对象一旦销毁，QPointer 自动变 null，调 isNull() 检查能避免野指针。

QThread::create(lambda) 创建一个新线程对象，新线程启动后会执行那个 lambda。

Lambda 捕获列表 [self, strategy, firmware, cancelFlag, total]：把 5 个变量按值复制到 lambda 内部，让 worker 线程也能访问它们。
为什么要 worker 线程？
strategy->flash() 里 DLL 的下载循环要发上千次 I2C 写命令，每次还要等串口响应（最长 5 秒）。一个 1MB 的固件可能要几秒到十几秒。这段时间如果跑在主线程，UI 会完全卡死，连进度条都不会动，取消按钮也点不动。

把烧录搬到 worker 线程，主线程继续刷新 UI，是 GUI 程序处理耗时任务的标准做法。
5 个 lambda 捕获分别为什么这么写？
  • self（QPointer）— 防 Service 在 worker 跑完前被销毁
  • strategy（裸指针）— Registry 持有它，生命期肯定够长
  • firmware（值拷贝）— QByteArray 是隐式共享的，零深拷贝
  • cancelFlag（shared_ptr 拷贝）— 引用计数 +1，确保 worker 用时不会被销毁
  • total（值拷贝）— 一个 int 而已

6. 把所有东西串起来

读到这里你已经看完所有核心代码。一次完整烧录的链路是：

  1. 用户在 UI 选好 IC 和文件，点"开始"
  2. UI 调 FwFlashService::startFlash()
  3. Service 通过策略名字从 Registry 找到具体的 AwSdkStrategy 实例（多态）
  4. Service 创建 worker 线程，把 strategy->flash() 放进去跑
  5. Worker 线程执行 AwSdkStrategy::flash()，跑 5 步 SDK 流程
  6. 下载步骤里 DLL 反复回调写 I2C — 通过 thunk + thread_local 路由到 onWriteI2c
  7. onWriteI2c 调 syncI2cWrite，异步包同步把数据通过串口发到 STM32，再到 IC
  8. 烧录完成后 worker 通过 invokeMethod 把结果送回主线程，UI 显示"完成"
