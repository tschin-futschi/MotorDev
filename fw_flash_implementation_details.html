1. 构造函数：AwSdkStrategy::AwSdkStrategy()

📄 aw_sdk_strategy.cpp（构造函数）
AwSdkStrategy::AwSdkStrategy(CommandDispatcher *dispatcher, LogSink logSink)
    : m_dispatcher(dispatcher)
    , m_logSink(std::move(logSink)) {
}

构造函数函数体是空的（{ } 里面什么都没写）。所有"工作"发生在那个奇怪的 : 后面那段，叫"初始化列表"。

初始化列表（member initializer list）：构造函数参数列表后面、函数体前面，用 : m_a(a), m_b(b) 这种形式给成员变量赋初值。

语法：
ClassName::ClassName(参数们)
    : 成员1(初值1)
    , 成员2(初值2)
    , 成员3(初值3) {
    // 函数体（可空）
}
每个成员单独一行，开头用 : 或 , 分隔。
为什么用初始化列表，而不是在函数体里赋值？

对比两种写法：

写法 A（函数体赋值）： 
AwSdkStrategy::AwSdkStrategy(CommandDispatcher *d, LogSink s) {
    m_dispatcher = d;          // 先默认构造，再赋值（两步）
    m_logSink = s;
}
写法 B（初始化列表）： 
AwSdkStrategy::AwSdkStrategy(CommandDispatcher *d, LogSink s)
    : m_dispatcher(d), m_logSink(std::move(s)) {}     // 一步直接初始化
  • 写法 A：成员先被默认构造（创建空对象），再被赋值，多一次操作
  • 写法 B：成员直接用参数构造，更高效
对内置类型（int、指针）影响不大，但对复杂类型（QString、std::function、QByteArray）就有性能差别。C++ 良好习惯：能用初始化列表就用初始化列表。

另外，const 成员、引用成员、没有默认构造函数的成员，必须用初始化列表。
std::move(logSink) 的作用：把 logSink 标记为"我不再需要了"，然后让 m_logSink 直接窃取它的内容（而不是复制）。

对 std::function 这种内部可能持有大对象（lambda 捕获了大堆变量）的类型，move 比拷贝快得多。

第一遍读规则：看到 std::move(x)，就把它理解成"把 x 高效地搬过来"。

2. 析构函数：~AwSdkStrategy()

📄 aw_sdk_strategy.cpp（析构函数）
AwSdkStrategy::~AwSdkStrategy() {
    if (m_library.isLoaded()) m_library.unload();
}

整段就一行：如果 DLL 当前还加载着，就卸载它。

QLibrary 是 Qt 提供的"动态库加载器"。Windows 下加载 .dll、Linux 下加载 .so，接口统一。

常用方法：
  • setFileName(path) — 设置要加载的文件
  • load() — 实际加载，成功返回 true
  • isLoaded() — 当前是否已加载
  • resolve(fnName) — 找到 DLL 里的某个导出函数，返回函数指针
  • unload() — 卸载
为什么要检查 isLoaded()？
AwSdkStrategy 对象创建时没立刻加载 DLL（构造函数没调 load），DLL 是在第一次 flash() 调用 ensureDllLoaded() 时才加载的。如果对象创建后从未被烧录过就被销毁，m_library 处于"未加载"状态，unload 在这种状态下行为未定义，所以加一道保险。

3. ensureDllLoaded() — DLL 加载与符号解析

这是 5 步流程的"第 0 步"——把 AW86100.dll 加载进进程，并把里面的 6 个导出函数地址保存到成员变量。

📄 aw_sdk_strategy.cpp（ensureDllLoaded，节选）
bool AwSdkStrategy::ensureDllLoaded(QString *errorMessage) {
    const QString name = dllPath();          // 子类返回 "AW86100.dll"
    m_library.setFileName(name);
    if (!m_library.load()) {
        // 加载失败：写错误信息并返回
        const QString reason = m_library.errorString();
        if (errorMessage) *errorMessage = QStringLiteral("Failed to load %1: %2").arg(name, reason);
        return false;
    }

第一段：调 m_library.load() 加载 DLL。失败的话用 Qt 的 QStringLiteral("...").arg(...) 拼出错误字串写到 errorMessage 出参。

QStringLiteral("...") 是 Qt 的字面量字符串构造，比 QString("...") 快（编译期生成）。%1、%2 是占位符，.arg(a, b) 把它们替换为 a、b。

3.1 用 lambda 包装"解析或失败"的重复逻辑

    auto resolveOrFail = [&](const char *fn, void **out) -> bool {
        QFunctionPointer p = m_library.resolve(fn);
        if (p == nullptr) {
            if (errorMessage) *errorMessage = QStringLiteral("DLL 缺少导出符号：%1").arg(fn);
            return false;
        }
        *out = reinterpret_cast<void *>(p);
        return true;
    };

这个 lambda 接收"函数名"和"用来接收地址的指针的指针"，去 DLL 里查这个名字，找到了就把地址写到 *out，找不到就报错返回 false。

函数指针 vs void*：

C++ 标准上，函数指针和数据指针（void*）是不同的东西，不能直接互转。但 Qt 的 QLibrary::resolve 返回的是 QFunctionPointer（一种通用函数指针类型），需要转成具体的函数指针类型才能调用。

reinterpret_cast<T>(x) 是 C++ 的"强制类型转换"，最霸道的一种 — 它告诉编译器"我不管你怎么想，把这个数据按 T 类型解释"。常用于：
  • 函数指针 ↔ void*
  • 不同类型的指针之间
  • 整数 ↔ 指针
用 reinterpret_cast 是危险的，但在与 C 库 / 操作系统接口对接时不可避免。

对比 C 风格 (T)x：C 风格的强制转换会自动选择各种 cast 中的一种，不明显 — C++ 推荐用具体名字 static_cast / reinterpret_cast / const_cast / dynamic_cast，让代码更显眼、更可读。
void **out 是什么？"指向指针的指针"。

为什么要这样？因为 lambda 要"修改外面的指针"。如果只传 void *out，lambda 内部 out = ... 改的是参数副本，外面看不到。所以传地址（指针的指针），*out = ... 才能改到外面那个指针。

调用时用 &p： 
void *p = nullptr;
resolveOrFail("AwOIS_ExtFuncInit", &p);   // 传指针 p 的地址

3.2 6 次解析（每个导出函数一次）

    void *p = nullptr;
    if (!resolveOrFail("AwOIS_ExtFuncInit", &p)) return false;
    m_fnInit = reinterpret_cast<FnExtFuncInit>(p);

    if (!resolveOrFail("Aw_OIS_ExtFuncDeInit", &p)) return false;
    m_fnDeInit = reinterpret_cast<FnExtFuncDeInit>(p);

    // ... 4 次类似 ...

    return true;
}

每次：resolveOrFail 拿到 void*，再 reinterpret_cast 成对应函数指针类型存到成员变量。这 6 个成员变量（m_fnInit / m_fnDeInit / m_fnBootcontrol / ...）就是后面 5 步流程实际调用的"DLL 函数把柄"。

为什么不直接 reinterpret_cast<FnExtFuncInit>(m_library.resolve("..."))？
那样做也对，但每次都要写一长串。提取成 lambda 后每次只需要一行，还能集中处理"找不到符号"的错误情况。抽取重复逻辑成 lambda 是 C++ 现代风格的常见做法。

4. flash() 主体 — 5 步流程的指挥棒

这部分前面的指南讲过框架，这里聚焦实现细节。

📄 aw_sdk_strategy.cpp（flash 函数主体，关键节选）
bool AwSdkStrategy::flash(...) {
    // 状态复位（每次烧录都从干净状态开始）
    m_cancelFlag = &cancelFlag;
    m_progress = std::move(progress);
    m_sentBytes = 0;
    m_lastReportedBytes = 0;
    m_lastReportTimeMs = QDateTime::currentMSecsSinceEpoch();
    m_inDownload = false;
    m_initCalled = false;
    m_bootEntered = false;

    s_currentInstance = this;            // thunk 路由用

    auto bail = [&](bool ok) -> bool {
        if (!ok) emergencyCleanup();
        s_currentInstance = nullptr;
        m_cancelFlag = nullptr;
        m_progress = nullptr;
        return ok;
    };
    // ... 5 步流程，每步前 checkCancel ...
}

关键点：

  1. 状态复位段：8 个成员变量被显式清零/置默认。这是防御性编程 — 即便上次烧录失败留下脏状态，本次也是干净开始。
  2. s_currentInstance = this：把"当前正在烧录的实例"记到 thread_local 静态变量。后面 DLL 通过 thunk 回调时才能找到这个实例。
  3. bail lambda：函数退出时统一调用，做收尾工作（清空指针、必要时调 emergencyCleanup）。
为什么要个 bail lambda 而不直接写收尾代码？
函数有 5 个失败退出点（5 步流程每步都可能失败），如果每个退出点都写一遍 3 行收尾，代码会冗余且容易写漏。bail 把"返回前必做的事"集中起来，每个退出点写 return bail(false) 就够。

这是 C 程序员常用的"goto cleanup"模式在 C++ 里的现代化等价表达。

5. 5 步具体实现：doInit / doBootcontrol / doDownload / doResetChip / doDeInit

5.1 doInit() — 注入 thunk 函数指针

bool AwSdkStrategy::doInit(QString *errorMessage) {
    DllExtFuncList list = {
        &AwSdkStrategy::extReadI2cThunk,
        &AwSdkStrategy::extWriteI2cThunk,
        &AwSdkStrategy::extOutputLogThunk,
    };
    const int ret = m_fnInit(&list);
    if (ret != 0) {
        // 失败处理
        return false;
    }
    m_initCalled = true;
    return true;
}
DllExtFuncList list = { ..., ..., ... }; 是聚合初始化（aggregate initialization）。如果一个结构体只有公开数据成员、没有自定义构造函数，可以用 { 值1, 值2, 值3 } 按声明顺序赋值。

DllExtFuncList 在头文件里定义为： 
struct DllExtFuncList {
    FnReadI2c   pFunc_I2C_Read;
    FnWriteI2c  pFunc_I2C_Write;
    FnOutputLog pFunc_OutputLog;
};
所以 {a, b, c} 按顺序填充 3 个成员。这是 C 风格语法，C++ 沿用。
&AwSdkStrategy::extReadI2cThunk 是"取一个静态成员函数的地址"。

对于静态成员函数，&ClassName::funcName 拿到的就是普通函数指针，类型与 FnReadI2c 等价，可以直接赋给 DllExtFuncList 的字段。

注意：如果是非静态成员函数，&ClassName::method 是成员函数指针，是另一种东西，不能直接当 C 函数指针用。这就是为什么 thunk 必须是 static 的。

m_fnInit(&list) 就是真正调 DLL 里的 AwOIS_ExtFuncInit 函数。m_fnInit 是函数指针，(&list) 是参数。

调用成功后设 m_initCalled = true，告诉 emergencyCleanup："如果失败回收，记得调对应的 DeInit"。

5.2 doDownload() — 真正烧录的关键

bool AwSdkStrategy::doDownload(QByteArray firmware, QString *errorMessage) {
    m_inDownload = true;                       // 标记进入下载段（让 onWriteI2c 累计进度）
    if (m_progress) m_progress(0);             // UI 进度条归零

    DllByte *buf = reinterpret_cast<DllByte *>(firmware.data());
    const DllUInt32 len = static_cast<DllUInt32>(firmware.size());
    const int ret = m_fnMoveBinDownload(buf, len, static_cast<DllInt>(kDLNumType_0));
    m_inDownload = false;                      // DLL 调用结束

    // 取消优先：DLL 即便返回失败，只要 cancelFlag 是 true 就归类为"用户取消"
    if (m_cancelFlag != nullptr && m_cancelFlag->load()) {
        if (errorMessage) *errorMessage = QStringLiteral("用户取消");
        return false;
    }
    if (ret != 0) {
        // 真失败
        return false;
    }
    return true;
}
static_cast<T>(x) 是相对"安全"的强制类型转换，主要用于：
  • 整数类型之间（如 int → unsigned int）
  • 浮点 ↔ 整数
  • 父类指针 ↔ 子类指针（向下转有运行时检查）
  • 显式调用类型转换运算符
比 reinterpret_cast 安全，但仍然是显式转换。

这里 firmware.size() 返回 qsizetype（可能是 long long），DllUInt32 是 32 位 unsigned。这种"缩窄转换"必须显式 cast，否则编译器警告。
reinterpret_cast<DllByte *>(firmware.data())：
QByteArray::data() 返回 char*，但 DLL 接口要求 unsigned char*（即 DllByte*）。两者底层都是字节，但类型不同。reinterpret_cast 强行解释。这是与 C 接口对接时的常见操作。
"取消优先"为什么这样写？
当用户取消时：
① cancelFlag 被置 true
② 后续 onWriteI2c 看到 cancelFlag 后返回非零
③ DLL 收到非零返回值，AwMoveBinDownload 自己中止并返回错误码

此时 ret != 0，但真实原因是用户取消，不是 DLL 失败。所以代码先看 cancelFlag，是 true 就归类成"用户取消"，这样 UI 显示的提示更准确。

其他 4 步（doBootcontrol / doResetChip / doDeInit）结构都一样：调 DLL 函数 → 检查返回值 → 失败则填错误信息。doInit / doBootcontrol 成功时还会翻转对应状态标志（m_initCalled / m_bootEntered），用于异常收尾决策。

6. 回调路由：3 个 thunk + 3 个 on 方法

前面指南详细讲过 thunk 模式。这里只展示一组实际代码。

// 静态 thunk — DLL 直接调这个
int AwSdkStrategy::extWriteI2cThunk(DllByte devId, DllByte wrSize, DllByte *wrData) {
    AwSdkStrategy *inst = s_currentInstance;
    if (inst == nullptr) return -1;
    return inst->onWriteI2c(devId, wrSize, wrData);
}

// 实例方法 — 真正干活
int AwSdkStrategy::onWriteI2c(DllByte devId, DllByte wrSize, DllByte *wrData) {
    if (m_cancelFlag != nullptr && m_cancelFlag->load()) return -1;
    const int rc = syncI2cWrite(devId, /*addrSize=*/0, /*addr=*/nullptr,
                                wrSize, wrData, kI2cTimeoutMs);
    if (rc == 0 && m_inDownload) {
        notifyProgress(static_cast<qint64>(wrSize));
    }
    return rc;
}

Thunk 体只有 3 行：从 thread_local 拿到当前实例 → 调实例方法。onWriteI2c 做 3 件事：检查取消、调同步 I2C、累计进度。

注释 /*addrSize=*/0 这种语法是给读者看的"命名参数提示"。函数调用时，看不到参数名只看到值，加这种行内注释可以让代码更易读。不是 C++ 语法，是注释而已。

7. syncI2cWrite() — 异步包同步详解

整个烧录代码最复杂的一段，但理解后就豁然开朗。

7.1 等待状态结构体

namespace {

struct I2cWaitState {
    QMutex mutex;
    QWaitCondition cond;
    bool received = false;
    bool isError = false;
    QByteArray data;
    QString errMsg;
};

}  // namespace
匿名 namespace（无名命名空间）：namespace { ... }。里面定义的东西只在当前 .cpp 文件可见。

类似 C 的 static 关键字（static 文件作用域），但更现代、更通用。常用于在 .cpp 里定义"仅本文件用"的辅助类型/函数，不污染其他文件。

这里 I2cWaitState 只在 syncI2cWrite 和 syncI2cRead 里用，没必要让其他文件看到。
struct 与 class 的区别：几乎一样，唯一区别是默认访问权限：
  • struct 默认 public
  • class 默认 private
C++ 习惯：纯数据（数据袋）用 struct，有行为/封装的用 class。I2cWaitState 就是个"等待响应时几个变量打包在一起"的数据袋，用 struct 合适。

这个结构装着等待响应需要的所有信息：互斥锁、条件变量、是否已收到、是否出错、数据、错误信息。

7.2 函数主体

int AwSdkStrategy::syncI2cWrite(uint8_t devId, ..., int timeoutMs) {
    if (m_dispatcher == nullptr) return -1;

    // ① 编码协议帧负载
    const QByteArray payload =
        MotorProtocol::encodeI2cTransferWrite(devId, addr, addrSize, data, dataLen);

    // ② 创建共享的等待状态
    auto state = std::make_shared<I2cWaitState>();

    // ③ 准备响应回调
    auto callback = [state](uint8_t cmd, uint8_t /*seq*/, const QByteArray &respData) {
        QMutexLocker locker(&state->mutex);
        state->received = true;
        if (cmd == MotorProtocol::CmdErrorResponse || ...) {
            state->isError = true;
            state->errMsg = ...;
        }
        state->cond.wakeOne();
    };

逐行讲：

  • ① MotorProtocol::encodeI2cTransferWrite(...) 把要发的内容（设备 ID、寄存器地址、数据）打包成协议帧字节流。这是协议层的工具函数，本文不展开。
  • ② 用 make_shared 在堆上创建一个 I2cWaitState 实例，state 是 std::shared_ptr<I2cWaitState>。
  • ③ 创建一个 lambda 作为响应到达时的回调。关键：lambda 按值捕获了 state — 这意味着 shared_ptr 引用计数 +1，确保 worker 拿到响应时 state 还活着。
为什么 state 用 shared_ptr，而不是直接放局部变量？
因为 callback lambda 会被发到主线程的 dispatcher 队列里。如果 worker 函数提前返回（比如超时），栈上的局部 I2cWaitState 会被销毁；可主线程 dispatcher 之后才执行 callback，访问已销毁的对象 → 崩溃。

shared_ptr 让 worker 和主线程 dispatcher 共享所有权：谁后用完谁负责销毁。这是跨线程对象的标配。

7.3 跨线程发请求

    QPointer<CommandDispatcher> disp(m_dispatcher);
    QMetaObject::invokeMethod(m_dispatcher, [disp, payload, callback]() {
        if (disp.isNull()) return;
        disp->submitCommand(MotorProtocol::CmdI2cTransferWrite, payload,
                             CommandDispatcher::High, callback);
    }, Qt::QueuedConnection);

把"调 dispatcher 发命令"这件事投递到主线程执行（因为 dispatcher 住在主线程，worker 不能直接调它）。lambda 按值捕获 disp / payload / callback，这三个对象会被复制到主线程使用。

7.4 worker 阻塞等待

    QMutexLocker locker(&state->mutex);
    while (!state->received) {
        if (!state->cond.wait(&state->mutex, static_cast<unsigned long>(timeoutMs))) {
            // 超时
            return -1;
        }
        if (m_cancelFlag != nullptr && m_cancelFlag->load()) return -1;
    }
    if (state->isError) return -1;
    return 0;
}
QMutexLocker locker(&mutex); 是 RAII 风格的自动锁。
构造时调 mutex.lock()，析构时（变量出作用域）自动调 mutex.unlock()。

这样写的好处：不用记得手动解锁。即使中间 return 提前退出，编译器保证 locker 对象被销毁，锁一定会被释放。

对比手写版： 
mutex.lock();
// ... 各种 return ...
mutex.unlock();   // 容易忘，或某条 return 路径漏掉 → 死锁
QWaitCondition::wait(&mutex, ms)：
调用时做 3 件事：① 释放 mutex；② 阻塞当前线程；③ 被唤醒后重新获取 mutex。

返回值：true = 被 wakeOne()/wakeAll() 唤醒；false = 超时。

代码 if (!state->cond.wait(...)) 等价于"如果是超时（不是被唤醒）"。
为什么是 while (!state->received) 不是 if？

因为 QWaitCondition::wait 可能"虚假唤醒"（spurious wakeup）— 没人真的 wakeOne，但 wait 自己醒来。这是底层操作系统 API 的特性，所有条件变量都有这毛病。

正确范式：用循环检查实际条件： 
while (!条件成立) {
    cond.wait(&mutex);   // 醒来后重新检查条件
}
记住：所有条件变量的等待都应该写在 while 循环里。

整个 wait 循环逻辑：

  • 每次被唤醒后看 state->received — 如果真的收到响应了，跳出循环
  • 如果没收到（虚假唤醒），继续等
  • 等的过程中如果超时，返回 -1
  • 等的过程中如果用户取消，也返回 -1

8. notifyProgress() — 节流策略

void AwSdkStrategy::notifyProgress(qint64 sentDelta) {
    m_sentBytes += sentDelta;
    const qint64 now = QDateTime::currentMSecsSinceEpoch();
    const bool intervalElapsed = now - m_lastReportTimeMs >= kProgressMinIntervalMs;
    const bool bytesAccumulated = m_sentBytes - m_lastReportedBytes >= kProgressMinBytes;
    if (sentDelta == 0 || intervalElapsed || bytesAccumulated) {
        if (m_progress) m_progress(m_sentBytes);
        m_lastReportedBytes = m_sentBytes;
        m_lastReportTimeMs = now;
    }
}

逻辑：累计字节数，满足以下任一才真正触发回调：

  1. sentDelta == 0 — 强制初始化（doDownload 开头会调一次以归零进度条）
  2. 距上次上报超过 16ms（kProgressMinIntervalMs）— 时间够久了，让 UI 走两步
  3. 字节数增量超过 1KB（kProgressMinBytes）— 累积够多了
为什么要节流？
DLL 在 AwMoveBinDownload 内部可能每写 64 字节就回调一次。一个 1MB 固件 = 1.6 万次回调。如果每次都触发 UI 更新，进度条会被狂刷，主线程事件队列爆炸。

16ms 对应 60Hz 刷新率，正好是人眼极限；1KB 是合理的最小可见进度增量。

9. emergencyCleanup() — 失败收尾

void AwSdkStrategy::emergencyCleanup() {
    if (m_bootEntered && m_fnResetChip != nullptr) {
        const int ret = m_fnResetChip();
        if (ret != 0) {
            log(LogLevel::Warn, QStringLiteral("收尾 AwResetChip 返回 %1（忽略）").arg(ret));
        }
        m_bootEntered = false;
    }
    if (m_initCalled) {
        doDeInit();
    }
}

失败 / 取消时被 bail 调用，根据 m_bootEntered / m_initCalled 两个状态标志决定收尾步骤：

失败发生在m_initCalledm_bootEntered需要收尾
DLL 加载阶段falsefalse什么都不用
doInit 失败falsefalse什么都不用
doBootcontrol 失败truefalse调 DeInit
doDownload 失败truetrue先 ResetChip 让 IC 退出 boot 模式，再 DeInit
为什么 ResetChip 失败时只记 warn 不返回错误？
因为收尾本身的失败不应该掩盖原始错误。比如下载真失败的原因是"DLL 写 I2C 超时"，这是用户需要看到的真原因。如果收尾时 ResetChip 又恰好失败，那把"ResetChip 失败"当成主错误返回会误导用户。所以收尾错误只记日志。

10. FwFlashService::startFlash + worker 创建

这部分前面指南已经详细讲过，这里只简单回顾要点。

📄 fwflashservice.cpp（关键节选）
void FwFlashService::startFlash(const QString &icModel, ...) {
    // 校验
    if (isBusy()) return;
    FlashStrategy *strategy = m_registry->find(icModel);
    if (strategy == nullptr) return;

    // 准备
    m_cancelFlag = std::make_shared<std::atomic<bool>>(false);
    setState(State::Preparing);

    // 进入主流程
    runPreflightAndFlash(strategy, firmware);
}

跳过前置 4 步停止序列后，进入 worker 线程创建：

    auto cancelFlag = m_cancelFlag;        // shared_ptr 拷贝（计数+1）
    const qint64 total = m_totalBytes;
    QPointer<FwFlashService> self(this);

    m_workerThread = QThread::create([self, strategy, firmware, cancelFlag, total]() {
        QString errorMsg;
        auto progress = [self, total](qint64 sent) {
            if (self.isNull()) return;
            emit self->progressUpdated(sent, total);
        };

        // 真正烧录在这里跑（worker 线程被阻塞几秒到十几秒）
        const bool ok = strategy->flash(firmware, progress, *cancelFlag, &errorMsg);

        // 结果回主线程
        QMetaObject::invokeMethod(self.data(), [self, ok, errorMsg]() {
            if (ok) self->setState(State::Completed);
            else    self->setState(State::Failed);
            emit self->finished(ok, ok ? "成功" : errorMsg);
        }, Qt::QueuedConnection);
    });

    connect(m_workerThread, &QThread::finished, m_workerThread, &QObject::deleteLater);
    m_workerThread->start();

关键流程：

  1. QThread::create(lambda) 创建一个新线程对象，启动后跑 lambda
  2. Lambda 按值捕获 5 个变量到新线程使用
  3. 新线程调 strategy->flash()（同步阻塞，几秒到十几秒）
  4. 结果通过 invokeMethod 回主线程切状态机
  5. connect(thread, finished, thread, deleteLater) — 线程结束后自动销毁线程对象
connect(发送方, 信号, 接收方, 槽) 是 Qt 信号槽机制的核心。当"发送方"发出指定信号时，自动调用"接收方"的指定槽。

&QThread::finished 是取信号地址（C++ 函数指针写法）。
&QObject::deleteLater 是取槽地址。

这一行的意思是："当 worker 线程完成时，让线程对象自己执行 deleteLater 销毁自己"——避免内存泄漏。
