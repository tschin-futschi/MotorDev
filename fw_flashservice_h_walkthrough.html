段 1（第 1–18 行）：文件头注释

// =============================================================================
// @file    fwflashservice.h
// @brief   固件烧录服务 — 协调前置序列、worker 线程、进度回调与状态机
//
// 职责：
// - 接收 UI 的 startFlash / cancelFlash 请求
// - 顺序触发前置停止序列（采样 → 发生器 → 循环写入 → PMIC），通过 4 个回调下发
// - 在独立 worker 线程内调用 FlashStrategy::flash()
// - 通过信号回 UI 上报状态、阶段文本、进度、日志、最终结果
// - 取消通过 std::atomic<bool> cancelFlag 协作式实现
//
// 线程：
// - 本类对象生存于主线程
// - flash() 调用在 QThread::create 创建的 worker 上执行
// - 所有信号 emit 跨线程时由 Qt 自动转 QueuedConnection
// - m_state 仅在主线程修改（worker 通过 invokeMethod 回主线程切换状态）
// =============================================================================

Doxygen 风格的文档注释。@file、@brief 是 Doxygen 标签。

这种"头注释把整个类的职责讲清楚"是良好工程习惯——后人读 .h 不需要看实现就能知道这个类是干什么的。特别注意"线程"段——明确写出"哪些操作在哪个线程"，是多线程类的必备文档。

段 5（第 33–34 行）：类声明开始 + Q_OBJECT

class FwFlashService : public QObject {
    Q_OBJECT

class FwFlashService : public QObject

声明类 FwFlashService，公开继承自 QObject。

为什么继承 QObject？因为只有 QObject 子类才能用 Qt 的信号槽机制。本类要 emit 信号给 UI，必须继承 QObject。

QObject 是 Qt 整个对象系统的根基类，提供：
  • 信号槽（signals/slots）
  • 父子对象树（parent-child）
  • 元对象信息（QMetaObject）
  • 跨线程调用（invokeMethod）

Q_OBJECT 宏

所有用到信号槽的 QObject 子类必须在类内的第一行写 Q_OBJECT 宏。它展开后会插入一堆代码，让 Qt 的"元对象编译器"（moc）能识别本类。
如果忘了写 Q_OBJECT，编译时连接器会报错： 
undefined reference to vtable for FwFlashService
或类似的难懂报错。新手常踩这个坑。

另外注意 Q_OBJECT 后没有分号——它是宏，展开后是一大段代码（带分号），所以这里写宏的时候不需要再加分号。

10 个状态

状态含义
Idle空闲，未开始烧录
Preparing准备中（参数校验后）
StoppingScope正在停采样
StoppingGenerator正在停信号发生器
StoppingCyclic正在停循环写入
DisablingPmic正在关 PMIC
Flashing真正烧录中
Completed已完成
Failed失败
Cancelled已取消

enum class 与 enum 的区别

// 旧式 enum
enum State { Idle, Preparing };
State s = Idle;          // 不需要前缀

// 新式 enum class(强类型枚举)
enum class State { Idle, Preparing };
State s = State::Idle;   // 必须加 State:: 前缀
强类型枚举（enum class）的好处：
  • 不会被隐式转换为整数（避免不小心当 int 用）
  • 不污染外部命名空间（多个 enum 可以有同名枚举值）
C++11 起推荐总是用 enum class，本项目遵循这个规范。

Q_ENUM(State)

Q_ENUM 把枚举注册到 Qt 元对象系统。注册后能：
  • qDebug() << state; 直接打印枚举名（输出 "State::Flashing" 而不是 "6"）
  • 信号槽可以传递这个枚举类型
  • QML / Qt Quick 可以使用
简单说：让 Qt "知道"这个枚举的存在。

explicit 关键字

explicit = 禁止隐式类型转换。

不加 explicit 时，编译器允许这种写法：

FwFlashService service = somePointer;   // ← 隐式从指针构造,容易出 bug

加了 explicit 后，只允许显式构造：

FwFlashService service(somePointer);     // ✓ 显式
FwFlashService service = somePointer;    // ✗ 编译错误
良好习惯：所有"单参数"或"前几个参数有默认值"的构造函数都该加 explicit。

参数 QObject *parent = nullptr —— Qt 对象树机制

Qt 对象树：每个 QObject 可以指定一个 parent，子对象在 parent 销毁时自动销毁——无需手动 delete。

= nullptr 是默认参数——不传时默认为 nullptr。

实际项目里：

// fwflashtab.cpp 第 101 行
m_service = new FwFlashService(m_registry.get(), this);
//                                                ^^^^
//                                                this 即 FwFlashTab,作为 parent

FwFlashTab 销毁时，m_service 会被自动销毁——这就是 Qt 对象树的好处。

~FwFlashService() override

析构函数。override 关键字表示"重写基类（QObject）的虚析构"。

段 10（第 69–73 行）：4 个回调注入接口

    /// @brief 注入"立即停止"回调；nullptr-safe，未设置时跳过该步骤（仅写日志）
    void setStopScopeCallback(StopCallback cb);
    void setStopGeneratorCallback(StopCallback cb);
    void setStopCyclicWriteCallback(StopCallback cb);
    void setDisablePmicCallback(StopCallback cb);

4 个 setter 函数，让外部注入"怎么停采样、怎么停发生器、怎么停循环写入、怎么关 PMIC"的回调。

注释说"nullptr-safe"——意味着不调用 set 也没事，相应步骤会跳过。

这是依赖反转（DI）模式：本类不知道有 ScopeService、GeneratorService 等存在，由外部（MainWindow）告诉它"停的时候调这个 lambda"。

好处：本类的代码不依赖 ScopeService 的头文件，编译解耦、测试方便。

实际注入：

// MainWindow 大致代码
fwFlashTab->setStopScopeCallback([scopeService]() {
    scopeService->requestStop();
});

State state() const { return m_state; }

inline 实现的成员函数：函数体直接写在 .h 里（一行），不需要 .cpp 里再实现一份。

这种"极简一行函数"放在 .h 里更高效——编译器可能直接 inline 替换，省去函数调用开销。

末尾的 const 表示"只读"——不修改对象状态。读取性函数都该加 const。

bool isBusy() const;

声明 + 函数体在 .cpp 里定义。也是 const 成员函数。

为什么不像 state() 一样写在 .h？因为 isBusy() 内部有逻辑（判断 6 个状态），实现稍复杂，放 .cpp 更清晰。

public slots: = Qt 特殊的访问控制——槽函数。本质上是普通成员函数，但能被信号 connect。

public slots 的访问权限和 public 一样——外部能直接调用。区别只是 Qt moc 工具会识别它们做特殊处理。

使用示例：

//                                              ^^^^^^^^^^^^^^^^^^^^^^^^^^^^
//                                              槽函数,响应按钮点击

signals: = Qt 特殊的访问控制——信号声明。

信号是"通知机制"，不是普通函数：
  • 信号没有函数体（moc 自动生成）
  • 用 emit signalName(args) 触发
  • 调用方"connect"它的对象会被通知

5 个信号的含义

信号含义UI 怎么响应
stateChanged状态机切换更新按钮使能
stageMessage阶段文字（"烧录中..."）更新阶段标签
progressUpdated进度条更新移动进度条
logMessage日志新增一行追加到日志面板
finished整体完成弹提示，恢复按钮

UI 端 FwFlashTab 把这 5 个信号 connect 到自己的 5 个槽函数。

为什么参数里出现 FwFlashService::State？按理说在类内可以直接写 State。但信号会被 moc 处理生成代码，生成的代码里需要"完整名字"才能正确识别枚举类型。Qt 文档推荐信号参数里的枚举写完整资格名。

整体回顾

class FwFlashService : public QObject {
    Q_OBJECT

public:
    [枚举] State (10 个状态)
    [枚举] LogLevel (4 级)
    [别名] StopCallback
    [构造/析构]
    [setter] 4 个回调注入接口
    [getter] state(), isBusy()

public slots:
    startFlash, cancelFlash       // ← UI 主动调用的入口

signals:
    stateChanged, stageMessage,    // ← 通知 UI 的信号
    progressUpdated, logMessage,
    finished

private:
    [实现] 4 个内部辅助函数
    [成员] 10 个私有变量
};

这是典型的 Qt 服务类结构：

  • 继承 QObject 用信号槽
  • public 段是对外接口
  • public slots 是主动入口
  • signals 是被动通知
  • private 是内部实现细节
