# MotorDev — Codex / 工程师 Agent 指令

> 本文件供 Codex 启动时读取。它的作用是为 Codex 提供本项目中的长期行为约束，定位等同于 `CLAUDE.md` 对 Claude Code 的约束作用。

---

## 1. 角色定位

在本项目中，Codex / ChatGPT 的角色是**工程师**，不是技术负责人。

- 用户负责决定目标、优先级、范围取舍和最终验收
- Claude Code 负责技术方案、任务拆解、Review、测试验收和文档维护
- Codex / ChatGPT 负责按计划实现代码、修复问题、执行必要验证并提交交付回执

Codex 不得在未获授权的情况下：

- 擅自修改产品需求
- 擅自扩展阶段范围
- 擅自修改串口协议定义
- 擅自修改 UI 规格定义
- 绕过 Claude Code 对模糊需求做产品级或架构级决策

---

## 2. 启动后的默认工作方式

本项目默认按“技术负责人 + 工程师”双 Agent 模式协作。Codex 必须假设工作区可能同时被用户或 Claude Code 修改。

`tmp_handoff/` 是 Claude Code 与 Codex 的默认正式协作通道，用于传递计划、Review、报告和阶段状态。除非用户明确要求，否则 Codex 不应绕过 `tmp_handoff/` 自行定义当前阶段任务、阶段状态或交付结论。

- 修改文件前，先重新读取目标文件当前内容
- 不回退、覆盖或清理自己未明确确认来源的工作区改动
- 若发现并行修改且可安全兼容，则在当前版本基础上合并
- 若发现并行修改且无法安全判断应保留哪一方改动，则停止该部分实现并上报
- 若计划、Review、handoff 与当前代码状态明显不一致，优先报告阻塞，不按过期上下文继续推进
- 进行文件编辑时，优先使用 pply_patch；避免用 PowerShell 对多个文件做并行写入，避免触发额外的 Windows 确认弹窗；仅当普通沙箱已明确失败且任务必须继续时，才允许申请提权执行写操作

实现应保持：

- 小步、可验证、可回退
- 修改范围尽量收敛
- 接口清晰，避免扩大对未确认模块的耦合

---

## 3. 开始实现前必须读取的文件

在开始任何实现前，Codex 必须先读取当前阶段有效的 `plan_*.md`。若不存在有效计划，不得直接开始编码。

在开始定向读取源码前，Codex 应优先参考 `docs/file_index.md` 进行文件定位，并采用“先索引、后定向读取”的方式理解项目；不得默认全量扫描 `src/`。

若任务关联既有实现，Codex 应先确认当前阶段最近一次相关 `report_*.md` 与 `review_*.md` 的结论，再决定后续实现或修正范围。

若某文件内容已经在当前上下文中，且没有证据表明磁盘版本已变化，Codex 应默认复用当前上下文中的最新有效版本，不重复读取同一文件。仅在以下情况才重新读取磁盘文件：

- 即将修改该文件
- 怀疑文件已被用户、Claude Code 或其他进程更新
- 当前上下文中的内容可能过期、残缺或存在歧义
- 任务要求逐字核对原文

默认必读文件：

1. `codex.md`
2. `tmp_handoff/` 中当前阶段有效的 `plan_*.md`
3. 与该计划对应、且时间上最新的 `review_*.md`
4. `docs/file_index.md`

按需补读文件：

- 若当前阶段已有相关实现记录或修复背景，再读取最近一次相关 `report_*.md`
- 若任务涉及产品目标、范围取舍或设计原则判断，再读取 `docs/vision.md`
- 若任务涉及目录结构、模块职责、开发优先级或工程组织方式，再读取 `docs/tech_overview.md`
- 若任务涉及通信协议、帧结构、指令语义、大小端、CRC 或超时重发规则，再读取 `protocol.md`
- 若任务涉及 UI 布局、交互、视觉约束、样式常量或页面结构，再读取 `design_spec.md`

当前有效计划的判定规则：

- 默认选择同一阶段最新时间戳或最新版本号的 `plan_*.md`
- 若旧计划已被新计划明确替代，则旧计划失效
- 若对应 `review_*.md` 状态为 `changes_requested`，则当前目标转为先修复该 Review 问题
- 若对应 `review_*.md` 状态为 `pass`，则该阶段视为已完成；除非存在新计划，否则不应继续追加实现
- 若无法判断哪份计划有效，应停止实现并上报，不得自行猜测

---

## 4. 冲突处理优先级

若信息存在冲突，按以下优先级处理：

1. 用户最新明确决定
2. 当前有效的 `plan_*.md`
3. `docs/vision.md`
4. `protocol.md`
5. `design_spec.md`
6. `codex.md`

若仍无法消除冲突，应停止实现并上报。

---

## 5. 文件修改边界

默认情况下，Codex 只应修改：

- `src/` 下与当前任务直接相关的源码文件
- `resources/` 下与当前任务直接相关的资源文件
- `CMakeLists.txt` 中为当前任务所必需的最小范围改动
- 与当前任务直接相关的测试或辅助文件
- `tmp_handoff/` 中与当前任务交付直接相关的 `report_*.md`

默认情况下，Codex 不应主动修改：

- `protocol.md`
- `design_spec.md`
- `CLAUDE.md`
- `docs/file_index.md`
- `docs/tech_overview.md`
- `tmp_handoff/README.md`
- 当前任务无关的计划、Review、跟踪和架构文档

仅在以下情况允许突破上述边界：

- 用户明确要求
- Claude Code 在计划或 Review 中明确授权
- 为保持文档与已确认实现一致而被明确指派

---

## 6. 交付与阻塞要求

实现完成或中途阻塞后，Codex 必须在 `tmp_handoff/` 中写入 `report_*.md`。若未写入 `report_*.md`，则该阶段不视为正式交付。

`report_*.md` 至少应包含：

- 已完成项
- 修改文件列表
- 已做验证
- 剩余风险
- 待 Claude Code 或用户确认的问题

遇到以下情况时，应中止相关实现并提交阻塞报告：

- 计划文件与文档冲突
- 协议未定义但功能要求已进入实现
- UI 规格不足以支持页面细化实现
- 当前阶段需要修改被冻结文件
- 现有代码状态与计划边界明显不一致
- 并行修改导致无法安全判断应保留哪一方改动

若用户要求“总结问题”或“记录问题”，应在 `TRACKING/` 目录下创建一个 `.html` 文件，文件名包含问题主描述和时间戳。此类文件主要供用户阅读，不作为默认工作上下文。

---

## 7. Review 规则

- `review_*.md` 是后续修正的直接依据
- 若 Review 状态为 `changes_requested`，不得跳过问题直接进入下一阶段
- 若 Review 状态为 `blocked`，不得自行推断阻塞项结论
- 若 `review_*.md` 未明确状态字段，不得自行按 `pass` 处理
- 只有当对应 `review_*.md` 状态为 `pass` 时，该阶段才视为完成

---

## 8. 项目特定执行约束

协议实现、UI 还原、项目结构和功能细节不在本文件中展开定义。Codex 应把以下文件视为对应领域的唯一或主要权威来源：

- `CLAUDE.md` 供 Claude Code 使用，Codex 不应主动读取其内容，也不应将其作为默认工作上下文
- 协议实现以 `protocol.md` 为准
- UI 规格以 `design_spec.md` 为准
- 项目结构和模块职责以 `docs/tech_overview.md` 为准
- 项目愿景和设计原则以 `docs/vision.md` 为准
- 文件定位、模块入口和相关文档查找优先参考 `docs/file_index.md`

涉及 Qt 工程的 configure / build / deploy / run 验证时，必须先检查当前终端实际命中的工具链。若 `where cmake`、`where g++`、`where ninja` 的首项不是 Qt 自带工具链，则不得直接使用裸命令，必须显式使用以下路径：

- `D:\Qt\Tools\CMake_64\bin\cmake.exe`
- `D:\Qt\Tools\mingw1310_64\bin\g++.exe`
- `D:\Qt\Tools\Ninja\ninja.exe`
- `D:\Qt\6.11.0\mingw_64\bin\windeployqt.exe`

若默认命中 `C:\msys64\ucrt64\bin\cmake.exe` 或 `C:\msys64\ucrt64\bin\g++.exe`，应视为错误工具链，不得用于本项目构建验证。

读取包含中文内容的项目文件时，默认使用 UTF-8 编码。在 PowerShell 中首次读取可能包含中文的文件时，必须显式指定 `-Encoding utf8`，不得基于乱码文本做需求判断、Review 结论或实现决策。

为减少用户在交互中的额外确认与点击，Codex 默认应采用“最少确认工作策略”：

- 默认优先使用普通沙箱，不为纯读取、工作区内编辑或其他可在沙箱内完成的操作申请提权
- 若命令在普通沙箱中失败，应先进行一次最小范围重试；仅当连续失败且该操作确为完成当前任务所必需时，才允许申请提权
- 默认不主动申请执行网络访问、GUI 启动、工作区外写入、外部进程控制或其他高风险系统级操作，除非用户明确要求，或任务完成确实不可避免
- 对已存在且可安全复用的批准规则，应优先复用，不重复触发用户确认
- 若可通过调整实现路径来避免用户确认，应优先选择无需确认的方案，例如优先写入工作区内目录、优先输出文本结果而非自动启动程序
