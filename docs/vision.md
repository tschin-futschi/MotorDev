# MotorDev — 项目愿景

> 版本：v0.2 | 日期：2026-05-06

---

## 一句话定位

MotorDev 是一款专业的电机驱动模组调试上位机，将寄存器读写、固件烧录和实时波形观测整合于一体，为电机调试提供高效、直观、可扩展的一站式工具体验。

---

## 我们要解决的问题

电机驱动模组的调试工作通常依赖多个分散工具——通用串口助手查寄存器、专用烧录器刷固件、示波器看波形。这种工作流存在以下痛点：

- **工具碎片化**：调试一个问题需要在多个工具间反复切换，效率低
- **上手成本高**：通用工具需要手动拼帧、查协议文档，容易出错
- **缺乏芯片适配**：通用工具不理解寄存器语义，无法提供有意义的显示和校验
- **数据断裂**：寄存器值、波形数据、烧录状态分散在不同工具中，难以关联分析

---

## MotorDev 的核心价值

| 价值 | 说明 |
|------|------|
| **一站式整合** | 寄存器读写、固件烧录、实时波形观测在同一个工具内完成，无需切换 |
| **深度适配** | 针对公司电机驱动模组的寄存器定义、通信协议做专属优化，开箱即用 |
| **专业且易用** | 面向工程师的专业功能，配合直观的 UI，降低调试门槛 |
| **可扩展架构** | 支持未来扩展更多芯片型号、通信方式和调试功能 |

---

## 目标用户

- **主要用户**：公司内部工程师（以项目负责人为主），用于日常电机驱动模组调试
- **潜在用户**：客户工程师，作为模组配套调试工具交付使用

---

## 当前阶段目标（v1.0）

构建功能完整的基础版本，围绕"调试电机驱动模组"主线，覆盖以下场景：

### 核心场景

1. **寄存器读写** — 可视化寄存器表格（4 组 × 20 行），支持单读/单写/批量读写、Hex/Dec 切换、配置自动持久化
2. **固件烧录** — 选择固件文件，一键烧录，进度可视，状态清晰（**仍为占位，待实现**）
3. **实时示波器** — 最多 8 通道波形显示，支持暂停/缩放/双击全屏/十字光标吸附读数；Overlay 与 Stacked 两种视图模式

### 配套场景（v1.0 一并交付）

为完成"调试电机驱动模组"的实际工作流，v1.0 同时交付以下与核心场景紧密耦合的能力：

4. **设备前置准备** — I2C 总线扫描、电机 IC 地址下发，作为后续读写/采样的前提
5. **PMIC 电源管理** — DRVDD / IOVDD / VCMVDD 三路电压配置 + 使能/禁用，支撑模组正常工作
6. **示波器辅助寄存器面板 + 循环写入** — 与示波器共面板的 8 行寄存器 R/W 与按周期循环写入，方便在观测波形时快速调参
7. **信号发生器** — Linear / Cosine / Sawtooth 三种模式，参数下发到 STM32 本地执行；锯齿波模式专用于验证「采样 → 示波器」链路完整性
8. **串口调试模拟器** — 在无硬件环境下模拟 STM32 全栈响应（含正弦波形流），用于上位机 UI/协议开发与端到端调试

### 运行质量基线

- 全局日志面板（实时 + 文件落盘）
- 启动画面、应用图标
- DPI 锁屏 / 多显示器边界稳定性
- 通信层心跳保活、超时重试、错误回退

### 技术约束

- 通信方式：串口（UART, 默认 460800）→ STM32 → I2C → 马达驱动芯片
- 目标平台：Windows 10/11 x64
- 技术栈：C++17 / Qt6 / CMake

## 演进策略

**先跑通，再做好。**

当前阶段专注于 v1.0 核心功能的完整交付，不做过早的扩展设计。等基础版本稳定运行后，再根据实际使用反馈决定后续方向。

---

## 设计原则

1. **结构化解耦** — 各层职责清晰隔离：UI 层、业务逻辑层、通信协议层、传输层各自独立。更换通信协议只需修改协议层代码，不影响 UI 和业务逻辑；更换传输方式（如串口换 USB）只需修改传输层，不影响上层
2. **实用优先** — 每个功能都必须解决真实的调试痛点，不做花哨但没用的东西
3. **稳定可靠** — 通信不丢帧，烧录不出错，工具本身不能成为调试的障碍
4. **快速上手** — 新用户 5 分钟内能完成第一次寄存器读取，无需翻文档
5. **易维护易扩展** — 新增功能（如新 Tab、新命令）应是局部添加而非全局改动；模块边界清晰，改一处不需要理解全局；代码可读性优先于技巧性
